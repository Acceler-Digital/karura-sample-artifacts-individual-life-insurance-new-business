# ドメイン共通要求仕様書

## 本書について

### 概要
本書は[ドメイン定義書](../domain-definition-document.md#一覧)で定義された各ドメインに対し、**3ドメイン以上で横断的に効く具体的な要求**を記載するドメイン要求仕様書です。

### 注記
本書では原則として How(具体的な実装手段)には踏み込みませんが、ビジネス上譲れない具体水準(例: 監査ログ保持期間・準拠すべき法的要件 等)はドメイン共通要求の不可分な構成要素として記載します。

| DOM-COMMON-FR-3 | 業務の中断・再開 | 意向把握・設計書作成・申込受付・告知受付 等、複数局面で発生する一時中断(顧客都合・募集人都合・端末環境都合)からの再開を、入力済みデータを失わずに行える共通の仕組みを提供する | 意向把握<br>設計書作成<br>申込受付<br>告知受付 を主とし、長時間業務全般に適用 |
| DOM-COMMON-FR-4 | 同意取得・同意管理 | 個人情報の取得・利用目的の明示と本人同意の取得、同意の撤回・更新を、プロダクト共通の枠組みで扱う。要配慮個人情報の取得時には法令上必要な同意を確実に取得する(法令面は §法務・コンプライアンス要求 DOM-COMMON-REG-3 と整合) | 意向把握<br>申込受付<br>告知受付<br>本人確認(KYC)<br>顧客情報管理 等 |

| DOM-COMMON-NFR-4 | 可用性 | 障害時の縮退運用方針 | 主要ドメイン(意向把握〜申込受付)単独で局所稼働可能とし、外部連携(KYC・反社・電子署名・決済)障害時の業務継続シナリオを保持する |
| DOM-COMMON-NFR-5 | 拡張性 | スケーラビリティ | クラウドネイティブを前提とし、募集チャネル拡大・新商品投入による申込件数増加に対し水平スケーリングで対応可能とする |
| DOM-COMMON-NFR-6 | 運用性 | 監視・障害検知 | 業務継続性に直結する主要処理(外部連携・契約成立判定・収納成立検知 等)の状態監視と異常検知を備える。監視対象はエラー率・レイテンシ・キュー滞留・外部連携応答・契約成立/収納成立の処理遅延 とし、閾値は SLO(稼働率99.9%・主要操作95パーセンタイル5秒)からバックキャストして設定する |

| DOM-COMMON-NFR-8 | 互換性 | 既存システム・外部サービスの更改耐性 | 契約管理・収納・帳票発送・本人確認・反社チェック・電子署名・決済 各サービスの更改・差し替えに対し、本プロダクト側の改修範囲を最小化する設計を志向する |
| DOM-COMMON-NFR-9 | 信頼性 | 外部連携・非同期処理のエラー検知・リトライ・冪等性 | 外部連携・非同期処理におけるエラー検知、再実行(リトライ)、二重実行防止(冪等性) のポリシーをプロダクト共通で定める。業務担当者が回復処理を業務上の手順で完結できるよう設計する |

## セキュリティ要求

### 共通要求

| ID | カテゴリ | 要求項目 | 内容・準拠基準 |
|---|---|---|---|
| DOM-COMMON-SEC-1 | 業界基準準拠 | FISC基準準拠 | FISC「金融機関等コンピュータシステムの安全対策基準・解説書(第11版)」に準拠する。基準改訂時は最新版へ追随する。FISC基準を本プロダクトの設計・運用の最上位準拠基準とする |
| DOM-COMMON-SEC-2 | 認証 | 認証方式 | 全直接利用者(ACT-1〜ACT-6)に対し、利用形態に整合する認証方式を提供する。要配慮個人情報を扱う操作には MFA を必須とする。社内利用者(ACT-1・ACT-3・ACT-4)は全社IdP(SAML/OIDC)と SSO 統合、代理店募集人(ACT-2)は代理店向けIdPとのフェデレーション、申込人/被保険者(ACT-5・ACT-6)は本人認証(メール/SMS OTP に本人確認連携を組み合わせ)とする |
| DOM-COMMON-SEC-3 | データ保護 | 通信時暗号化 | プロダクトの全通信を TLS 1.2 以上(TLS 1.3 を推奨)で暗号化する。TLS 1.1 以下、および FISC・業界実績で許容されない弱い暗号スイートは無効化・除外する |
| DOM-COMMON-SEC-4 | データ保護 | 保存時暗号化 | 個人情報・要配慮個人情報・業務上機密データはすべて保存時暗号化を必須とする(方式は AES-256)。鍵はクラウド KMS/HSM で集中管理し、職務分掌に基づくアクセス制御と年1回以上の鍵ローテーションを行う |
| DOM-COMMON-SEC-5 | アクセス制御 | RBAC・最小権限 | 役割・組織・商品・募集チャネル を組み合わせた RBAC を適用する。要配慮個人情報へのアクセスは職務上必要な担当者に限定する。特権アクセスは別系統で管理する |
| DOM-COMMON-SEC-6 | 監査ログ | ログ対象・改ざん不能性 | 個人情報・契約データへのアクセス・操作・参照を改ざん不能な形で記録する。ログ自体への改変も全件追跡可能とする |
| DOM-COMMON-SEC-7 | 監査ログ | 保存期間 | 法令(個人情報保護法・電子帳簿保存法 等)の保存義務および社内規程と整合する期間として、監査ログ・契約関係の証跡を10年間保持する |
| DOM-COMMON-SEC-8 | 脆弱性対策 | 第三者セキュリティ診断 | リリース前の第三者セキュリティ診断を必須化する。重大指摘の解消をリリース判定の必須条件とする |
| DOM-COMMON-SEC-9 | インシデント対応 | エスカレーション | 個人情報漏洩・不正アクセス・外部攻撃 等の検知時に、CSIRT・コンプライアンス部・経営層・(必要時)監督官庁・本人 へのエスカレーション手順を業務プロセスとして整備する |

## 受け入れ基準

本書記載の共通要求が満たされていることの検収観点を以下に示します。プロダクト全体としてのローンチ可否判断基準は [PRD §受け入れ基準](../product-requirement-document.md#受け入れ基準) を、個別機能・個別ドメインの受け入れ基準は各ドメイン要求仕様書 を参照してください。

* **横断機能要求の充足**: §機能要求(`DOM-COMMON-FR-*`)に挙げた業務通知・帳票出力・業務の中断/再開・同意取得 がプロダクト共通の仕組みとして実装され、各ドメインから利用可能であること
* **非機能要求の充足**: §非機能要求(`DOM-COMMON-NFR-*`)に挙げた性能・可用性・拡張性・運用性・保守性・互換性・信頼性 の目標値を業務時間帯・繁忙期相当のシナリオで検証済みであること
* **セキュリティの充足**: §セキュリティ要求(`DOM-COMMON-SEC-*`・`DOM-COMMON-SEC-DATA-*`)が第三者セキュリティ診断で重大指摘ゼロの状態に到達し、FISC基準準拠確認が完了していること
* **法令遵守**: §法務・コンプライアンス要求(`DOM-COMMON-REG-*`)の各遵守事項が業務プロセスに組み込まれ、UATで遵守状況を確認済みであること
* **UI/UX 方針の浸透**: §UI/UX 方針(`DOM-COMMON-UX-*`)が下流の画面コンセプト集・画面要件群・各ドメイン要求仕様書の画面関連記述に反映され、矛盾なく一貫していること

# プロダクト要求仕様書

## 本書について
### 概要
本書は、Sample生命保険株式会社 個人保険新契約システム(以下、本プロダクト)に対するプロダクト要求を記載したドキュメントです。

上流のビジネス要件定義書が「なぜやるのか(Why)」と「事業として何を実現するのか(事業レベルの What)」を扱うのに対し、本書は　**「プロダクトとして何を提供すべきか(プロダクトレベルの What)」** を扱います。具体的な機能・画面・実装手段(How)にはまだ踏み込まず、プロダクトとしての提供形態・利用体験・横断的な機能/非機能/制約を方針レベルで定義します。

また本書は **ドメイン要求仕様書群の親文書** としても機能し、ドメイン横断で共通して効く要求・方針レベルの要求 を集約します。各ドメイン要求仕様書は本書の共通要求を継承し、その上に個別ドメイン固有の業務ルール・業務状態遷移・業務運用(イレギュラー対応) 等を積み上げる構造となります。

### 想定読者
* プロジェクトマネージャー(PM)
* PdM
* ドメインエキスパート
* 上流成果物作成者
* ステークホルダー

### 注記
本書では原則として How(具体的な実装手段)には踏み込みませんが、**ビジネス・規制上譲れない具体水準(可用性目標値・監査ログ保持期間・準拠すべき業界基準 等)** はプロダクト要求の不可分な構成要素として本書で確定します。これらは「実装手段の選択」ではなく「実装が満たすべき水準・基準」であり、本書のスコープに含まれます。

## プロダクト概要

本プロダクトは、Sample生命保険の個人保険(終身・定期・医療・がん 等)における新契約手続きを、意向把握から保険証券発行に至るまで一気通貫で電子化する業務プロダクトです。意向把握・設計書作成・申込受付・告知受付・引受査定・第一回保険料収納・契約成立(計上)・保険証券発行 までを単一のプロダクト体験として提供します。

提供形態は、社内営業職員・代理店募集人による対面/非対面のハイブリッド募集を主とし、引受査定担当者・新契約事務担当者の社内業務、および申込人・被保険者本人による告知・本人確認入力を包含します。契約成立後の保全業務・継続保険料収納・保険金支払業務・帳票物理発送 等は本プロダクトの直接対象とせず、既存契約管理システム・収納システム・帳票発送システム との連携で完結させます。

将来的にフェーズ2以降で盛り込む、ダイレクト(顧客直接申込)・銀行窓販・組込型保険(エンベデッド・インシュアランス)等のチャネル拡張、新商品・特約追加、引受査定の機械学習活用 等の発展に耐える、チャネル非依存・商品拡張可能・外部疎結合な業務基盤を目指します。

## 体験設計

本章では、本プロダクトとして提供する価値・体験方針と、その価値を届ける対象となるシステムの利用ユーザーを、この順序で定義します。下流の体験設計成果物(ユーザーストーリー一覧・ユースケース一覧・画面要件群)を駆動する前提として整理します。

### 提供価値・体験方針

本プロダクトの提供価値は、紙・対面前提の現行新契約プロセスを、関係者全員にとっての業務上のストレスを下げるかたちで電子化することにあります。ビジネス要件定義書で示された事業戦略・KPI・将来拡張性 を踏まえ、下流の体験設計を駆動する設計原則として、以下の柱を置きます。

* **エンドツーエンドのリードタイム短縮**: 意向把握から契約成立までを単一プロダクト内で連結し、工程間の手戻り・データ転記・紙書類のやり取りを生じさせない。前工程の成果物(意向情報・設計書・申込情報・告知情報)を後工程が参照可能な形で引き継ぐことを設計原則とする
* **不備の発生抑止**: 紙起因の記入不備をなくすため、入力時点でのチェック(必須項目・整合性・関連項目間の矛盾)と前工程からの引き継ぎ精度の双方で不備の発生源を絶つ。引用: ビジネス要件定義書 - KPI「不備返戻率5%以下(現行比60%削減)」
* **引受査定の標準化と透明性**: 自動判定を主、医的査定を例外とする運用を前提に、判定ロジックの透明性・判定根拠の追跡可能性・段階的チューニング可能性を担保する。引用: ビジネス要件定義書 - KPI「自動判定率70%以上」
* **募集コンプライアンスの証跡化**: 意向把握・情報提供・適合性原則・告知の妥当性 等の募集コンプライアンス上の確認事象を、業務局面ごとに改ざん不能な形で証跡化する。お客様本位の業務運営原則(フィデューシャリー・デューティー)の業務プロセスへの埋め込みを基本姿勢とする
* **マルチチャネル展開耐性(フェーズ2以降)**: 本プロジェクト(初期フェーズ)の対象は営業職員チャネル・代理店チャネルとし、ダイレクト・銀行窓販・組込型保険 等のチャネル追加に耐える、チャネル非依存の業務処理基盤として設計する。チャネル別の体験差は許容しつつ、業務処理本体は共通化する
* **疎結合なエコシステム連携**: 既存内部システム(契約管理・収納・帳票発送)および外部サービス(本人確認・反社チェック・電子署名・決済)との連携は疎結合を原則とし、各システムの更改・差し替えに対する本プロダクトの改修範囲を限定する

#### 共通要求
| ID | カテゴリ | 要求項目 | 内容・準拠基準 |
|---|---|---|---|

| PRD-SEC-2 | 認証 | 認証方式 | 全直接利用者(ACT-1〜ACT-6)に対し、利用形態に整合する認証方式を提供する。要配慮個人情報を扱う操作には MFA を必須とする。社内利用者(ACT-1・ACT-3・ACT-4)は全社IdP(SAML/OIDC)と SSO 統合、代理店募集人(ACT-2)は代理店向けIdPとのフェデレーション、申込人/被保険者(ACT-5・ACT-6)は本人認証(メール/SMS OTP に本人確認連携を組み合わせ)とする |

| PRD-SEC-4 | データ保護 | 保存時暗号化 | 個人情報・要配慮個人情報・業務上機密データはすべて保存時暗号化を必須とする(方式は AES-256)。鍵はクラウド KMS/HSM で集中管理し、職務分掌に基づくアクセス制御と年1回以上の鍵ローテーションを行う |
| PRD-SEC-5 | アクセス制御 | RBAC・最小権限 | 役割・組織・商品・募集チャネル を組み合わせた RBAC を適用する。要配慮個人情報へのアクセスは職務上必要な担当者に限定する。特権アクセスは別系統で管理する |
| PRD-SEC-6 | 監査ログ | ログ対象・改ざん不能性 | 個人情報・契約データへのアクセス・操作・参照を改ざん不能な形で記録する。ログ自体への改変も全件追跡可能とする |
| PRD-SEC-7 | 監査ログ | 保存期間 | 法令(個人情報保護法・電子帳簿保存法 等)の保存義務および社内規程と整合する期間として、監査ログ・契約関係の証跡を10年間保持する |
| PRD-SEC-8 | 脆弱性対策 | 第三者セキュリティ診断 | リリース前の第三者セキュリティ診断を必須化する。重大指摘の解消をリリース判定の必須条件とする |
| PRD-SEC-9 | インシデント対応 | エスカレーション | 個人情報漏洩・不正アクセス・外部攻撃 等の検知時に、CSIRT・コンプライアンス部・経営層・(必要時)監督官庁・本人 へのエスカレーション手順を業務プロセスとして整備する |

#### データアクセス要求

本項では主要なデータエンティティを機密区分ごとに棚卸しし、各データのアクセス制御方針を確定します。ここで定めた機密区分は、上記セキュリティ要求(暗号化・RBAC・監査ログ 等)の適用範囲を駆動する入力であり、あわせて §「法務・コンプライアンス要求」(個人情報保護法)および下流の D2 データモデル定義からも参照されます。

| ID | エンティティ | 取扱区分 | 主な保有理由 | アクセス制御の方針 |
|---|---|---|---|---|
| PRD-SEC-DATA-1 | 顧客情報(申込人・被保険者・受取人の属性・関係性) | 個人情報 | 申込手続き・契約成立後の契約管理引き渡しに必須 | 役割×組織×目的に基づく最小権限制御。参照は監査ログ対象 |
| PRD-SEC-DATA-2 | 告知情報(健康状態・診査結果・職業 等) | 要配慮個人情報 | 引受査定の判定根拠 | 引受査定担当者・限定された業務担当者のみ参照可。参照は監査ログ必須・改ざん不能保存 |
| PRD-SEC-DATA-3 | 申込・契約情報(プラン内容・保険料・払込方法・受取人指定 等) | 個人情報・業務上機密 | 申込意思の確定・契約成立 | 役割×組織での権限制御。募集人は自身が関与する申込のみアクセス可 |
| PRD-SEC-DATA-4 | 商品仕様・引受基準・保険料計算ルール | 業務上機密 | 商品提案・引受判定の業務ルール源 | 業務部門による更新権限。参照は広範に許容、更新は限定 |
| PRD-SEC-DATA-5 | 募集人・代理店登録情報 | 業務上機密(法令義務) | 保険業法上の募集人登録義務、商品×募集人の権限制御 | 厳格管理。登録の正確性・最新性を最優先 |
| PRD-SEC-DATA-6 | 募集コンプライアンス証跡(意向把握・情報提供・適合性確認・告知・電子署名 等) | 個人情報含む・業務上機密 | コンプライアンス部・監督官庁・内部監査への説明 | 改ざん不能保存。参照は限定・全件監査ログ対象 |
| PRD-SEC-DATA-7 | 監査ログ・操作ログ | 業務上機密 | 内部統制・第三者セキュリティ診断・電子帳簿保存法対応 | 改ざん不能保存。保存期間は10年間(PRD-SEC-7 と整合) |

| スケジュール | 2025年12月の本番リリース(募集チャネル別の段階リリース)、2025年11月のUAT(段階実施)、2025年3月末のPRD承認、2025年5月末の外部連携IF仕様確定、2026年1月の並行運用期間終了 | 本書で確定すべき要求の量と粒度を「PRD承認時点で合意可能な水準」に絞り込み、PRD承認後は変更管理プロセスで詳細化する前提を取る |

## 受け入れ基準

プロダクト全体としてのローンチ可否判断基準を以下に示します。個別機能・個別ドメインの受け入れ基準は下流の各成果物(ドメイン要求仕様書・機能一覧・テスト計画 等)に降ろし、本書では「これが満たされなければプロダクトとしてローンチしない」レベルの統合的な基準を示します。

* **KPI達成見込み**: BRDの主要KPI(リードタイム平均5営業日以内・不備返戻率5%以下・自動判定率70%以上・稼働率99.9%以上・重大インシデント年0件・募集人NPS+10ポイント)を、ローンチ後に達成可能と見込める設計・運用が完了していること
* **法令遵守**: 保険業法・保険法・個人情報保護法・犯罪収益移転防止法・電子帳簿保存法・金融庁監督指針 の各遵守事項を業務プロセスに組み込み、UATで遵守状況を確認済みであること
* **セキュリティ**: 第三者セキュリティ診断において重大指摘がゼロであること、および FISC基準準拠の確認が完了していること
* **外部連携**: 主要外部連携(PRD-EXT-1〜PRD-EXT-7)について、正常時挙動・障害時挙動・縮退運用シナリオ・障害復旧訓練 が完了していること
* **業務継続性**: 月末月初繁忙期相当の処理量に対する性能・安定性検証が完了していること
* **並行運用**: リリース後の並行運用期間(2025年12月〜2026年1月)を通じて、現場の重大インシデント・業務停止・コンプライアンス違反 が発生しないこと
* **コンプライアンス監査対応**: コンプライアンス部・内部監査部 によるリリース前レビューでの重大指摘がゼロであること

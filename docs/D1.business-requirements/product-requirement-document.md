# プロダクト要求仕様書

## 本書について
### 概要
本書は、Sample生命保険株式会社 個人保険新契約システム(以下、本プロダクト)に対するプロダクト要求を記載したドキュメントです。

上流のビジネス要件定義書が「なぜやるのか(Why)」と「事業として何を実現するのか(事業レベルの What)」を扱うのに対し、本書は　**「プロダクトとして何を提供すべきか(プロダクトレベルの What)」** を扱います。具体的な機能・画面・実装手段(How)にはまだ踏み込まず、プロダクトとしての提供形態・利用体験・横断的な機能/非機能/制約を方針レベルで定義します。

また本書は方針・概念・体験設計・制約・受け入れ基準のみを扱い、ドメイン横断で共通する具体的要求は独立成果物の [ドメイン共通要求仕様書](./domain-requirements/common-requirement-document.md) に、ドメイン固有の業務ルール等は各ドメイン要求仕様書(個別)に降ろします(詳細は §「[ドメイン共通要求への分割方針](#ドメイン共通要求への分割方針)」参照)。

### 想定読者
* プロジェクトマネージャー(PM)
* PdM
* ドメインエキスパート
* 上流成果物作成者
* ステークホルダー

### 注記
本書では原則として How(具体的な実装手段)には踏み込みませんが、**ビジネス・規制上譲れない具体水準(可用性目標値・監査ログ保持期間・準拠すべき業界基準 等)** はプロダクト要求の不可分な構成要素として本書で確定します。これらは「実装手段の選択」ではなく「実装が満たすべき水準・基準」であり、本書のスコープに含まれます。

## プロダクト概要

本プロダクトは、Sample生命保険の個人保険(終身・定期・医療・がん 等)における新契約手続きを、意向把握から保険証券発行に至るまで一気通貫で電子化する業務プロダクトです。意向把握・設計書作成・申込受付・告知受付・引受査定・第一回保険料収納・契約成立(計上)・保険証券発行 までを単一のプロダクト体験として提供します。

提供形態は、社内営業職員・代理店募集人による対面/非対面のハイブリッド募集を主とし、引受査定担当者・新契約事務担当者の社内業務、および申込人・被保険者本人による告知・本人確認入力を包含します。

本プロダクトは内製スコープを意向把握 〜 保険証券発行 に絞り、以下の領域は **外部システム/外部サービス** に委ねます(本プロダクトは API/データ連携で参照する立場をとります):

- **既存内部システム(自社別システム)**: 商品マスタ管理システム(商品仕様・特約・引受基準・保険料計算ロジック)、募集人管理システム(募集人・代理店登録・募集権限)、契約管理システム(契約成立後の保全)、収納システム(第2回目以降の継続収納)、帳票発送システム(物理帳票の印刷・封入・郵送)
- **外部サービス**: 本人確認(eKYC)、反社チェック、電子署名、決済(第一回保険料の収納手段)

これらは [ドメイン共通要求仕様書 §「外部連携要求」](./domain-requirements/common-requirement-document#外部連携要求) で連携方針(疎結合性・縮退運用・冪等性・証跡保持)を具体化します。

将来的にフェーズ2以降で盛り込む、ダイレクト(顧客直接申込)・銀行窓販・組込型保険(エンベデッド・インシュアランス)等のチャネル拡張、新商品・特約追加、引受査定の機械学習活用 等の発展に耐える、チャネル非依存・商品拡張可能・外部疎結合な業務基盤を目指します。

## 体験設計

本章では、本プロダクトとして提供する価値・体験方針と、その価値を届ける対象となるシステムの利用ユーザーを、この順序で定義します。下流の体験設計成果物(ユーザーストーリー一覧・ユースケース一覧・画面要件群)を駆動する前提として整理します。

### 提供価値・体験方針

本プロダクトの提供価値は、紙・対面前提の現行新契約プロセスを、関係者全員にとっての業務上のストレスを下げるかたちで電子化することにあります。ビジネス要件定義書で示された事業戦略・KPI・将来拡張性 を踏まえ、下流の体験設計を駆動する設計原則として、以下の柱を置きます。

* **エンドツーエンドのリードタイム短縮**: 意向把握から契約成立までを単一プロダクト内で連結し、工程間の手戻り・データ転記・紙書類のやり取りを生じさせない。前工程の成果物(意向情報・設計書・申込情報・告知情報)を後工程が参照可能な形で引き継ぐことを設計原則とする
* **不備の発生抑止**: 紙起因の記入不備をなくすため、入力時点でのチェック(必須項目・整合性・関連項目間の矛盾)と前工程からの引き継ぎ精度の双方で不備の発生源を絶つ。引用: ビジネス要件定義書 - KPI「不備返戻率5%以下(現行比60%削減)」
* **引受査定の標準化と透明性**: 自動判定を主、医的査定を例外とする運用を前提に、判定ロジックの透明性・判定根拠の追跡可能性・段階的チューニング可能性を担保する。引用: ビジネス要件定義書 - KPI「自動判定率70%以上」
* **募集コンプライアンスの証跡化**: 意向把握・情報提供・適合性原則・告知の妥当性 等の募集コンプライアンス上の確認事象を、業務局面ごとに改ざん不能な形で証跡化する。お客様本位の業務運営原則(フィデューシャリー・デューティー)の業務プロセスへの埋め込みを基本姿勢とする
* **マルチチャネル展開耐性(フェーズ2以降)**: 本プロジェクト(初期フェーズ)の対象は営業職員チャネル・代理店チャネルとし、ダイレクト・銀行窓販・組込型保険 等のチャネル追加に耐える、チャネル非依存の業務処理基盤として設計する。チャネル別の体験差は許容しつつ、業務処理本体は共通化する
* **疎結合なエコシステム連携**: 既存内部システム(契約管理・収納・帳票発送)および外部サービス(本人確認・反社チェック・電子署名・決済)との連携は疎結合を原則とし、各システムの更改・差し替えに対する本プロダクトの改修範囲を限定する

## ドメイン共通要求への分割方針

複数ドメインに横断的に効く具体的要求事項は、本書から分離し独立成果物として [ドメイン共通要求仕様書](./domain-requirements/common-requirement-document.md) に集約します。各ドメイン要求仕様書(個別)はドメイン共通要求仕様書の要求を継承しつつ、ドメイン固有の業務ルール等を積み上げる関係になります。

| 章 | 含まれる要求カテゴリ | ID 体系 |
|---|---|---|
| §機能要求 | 業務通知・帳票出力・業務の中断/再開・同意取得 等の横断業務機能 | `DOM-COMMON-FR-*` |
| §非機能要求 | 性能・可用性・拡張性・運用性・保守性・互換性・信頼性 | `DOM-COMMON-NFR-*` |
| §外部連携要求 | 既存内部システム・外部サービスとの連携方針 | `DOM-COMMON-EXT-*` |
| §UI/UX 方針 | 対象デバイス・アクセシビリティ・トーン&マナー・エラー時体験 等 | ID なし(観点ベースの方針宣言) |
| §セキュリティ要求 / §データアクセス要求 | FISC準拠・認証・暗号化・RBAC・監査ログ・データエンティティ機密区分 等 | `DOM-COMMON-SEC-*` / `DOM-COMMON-SEC-DATA-*` |
| §法務・コンプライアンス要求 | 適用法令・規制 | `DOM-COMMON-REG-*` |

各ドメイン要求仕様書(個別)は、上記の `DOM-COMMON-*` を「継承する共通要求」として参照し、ドメイン固有の業務ルール(`DOM-<ドメインコード>-BR-*`)・業務運用イレギュラー(`DOM-<ドメインコード>-IRR-*`)を積み上げる構造になります。

## 受け入れ基準

プロダクト全体としてのローンチ可否判断基準を以下に示します。個別機能・個別ドメインの受け入れ基準は下流の各成果物(ドメイン要求仕様書・機能一覧・テスト計画 等)に降ろし、本書では「これが満たされなければプロダクトとしてローンチしない」レベルの統合的な基準を示します。

* **KPI達成見込み**: BRDの主要KPI(リードタイム平均5営業日以内・不備返戻率5%以下・自動判定率70%以上・稼働率99.9%以上・重大インシデント年0件・募集人NPS+10ポイント)を、ローンチ後に達成可能と見込める設計・運用が完了していること
* **法令遵守**: 保険業法・保険法・個人情報保護法・犯罪収益移転防止法・電子帳簿保存法・金融庁監督指針 の各遵守事項を業務プロセスに組み込み、UATで遵守状況を確認済みであること
* **セキュリティ**: 第三者セキュリティ診断において重大指摘がゼロであること、および FISC基準準拠の確認が完了していること
* **外部連携**: 主要外部連携(DOM-COMMON-EXT-1〜DOM-COMMON-EXT-9)について、正常時挙動・障害時挙動・縮退運用シナリオ・障害復旧訓練 が完了していること
* **業務継続性**: 月末月初繁忙期相当の処理量に対する性能・安定性検証が完了していること
* **並行運用**: リリース後の並行運用期間(2025年12月〜2026年1月)を通じて、現場の重大インシデント・業務停止・コンプライアンス違反 が発生しないこと
* **コンプライアンス監査対応**: コンプライアンス部・内部監査部 によるリリース前レビューでの重大指摘がゼロであること

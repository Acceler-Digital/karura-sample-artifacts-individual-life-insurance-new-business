## 図の方針(Mermaid `flowchart`・C4風)

- **可読性優先で `flowchart LR` の3カラム構成**にする: **利用者(左)→ 対象システム(中央)→ 外部システム(右)**。C4 専用記法(`C4Context`)は要素が増えると中央へ線が放射状に集中し可読性が落ちるため用いない。
- **色分けで C4 の意味を保つ**: `classDef` で 利用者=青系 / 対象システム=濃い青 / 外部システム=灰系。各ノードラベルに ID(`ACT-*` / `EXT-*`)を併記する。
- **線の向きを揃えて交差を減らす**: 利用者は全て `利用者 --> システム`、外部システムは全て `システム --> 外部システム`。逆向きの関係を作らない(例: 受け取り系も利用者起点の向きに統一する)。
- **`subgraph` でグルーピング**: 左を社内/社外、右を自社別システム/外部サービス 等で括ると見通しが良い。
- 関係ラベルは **業務言語の概括**(例: `引受査定` / `本人確認依頼`)。
- **L1 の抽象度を厳守**: 内部コンテナ・技術スタック・API/プロトコル名・データ項目は書かない(C4 L2・D3 設計へ降ろす)。

## 描く対象の範囲(本図のスコープ)

- 本図は **システムに直接情報を授受するアクター・外部システムのみ** を描く(全アクターの網羅ではない)。モニタリング・管理・監督・サポート等で**間接的にしか関与しない**業務関係者(受取人・各種募集管理者・経営層・IT部門・募集人サポート・監督官庁 等。システムを直接操作・連携しない者)は **載せない**。
- この範囲(直接関与のみ・全アクター網羅ではない旨)を**成果物本文と登場要素節の双方に明記する**。

## 登場要素の選定と各列の埋め方

**選定**:

- **人(利用者)**: システムを直接操作・連携するアクターのみを載せる。直接利用アクター(`ACT-1`〜`ACT-6`)、および統制・運用でシステムを直接操作するアクター(権限・基準・例外統制の業務部門責任者、目視判断・証跡提示を行うコンプライアンス部、監査証跡を検証する内部監査部、監査ログ監視の CSIRT 等)が対象。システムを直接操作せず報告・受領・監督に留まる者(監督官庁 等)は載せない(§描く対象の範囲)。
- **外部システム**: 外部システム一覧の `EXT-*` をそのまま引く(本書では外部システムを定義・採番しない)。`EXT-*` は原則すべて登場させる。
- **対象システム**: PRD §プロダクト概要の内製スコープ(意向把握〜保険証券発行を一気通貫で電子化する業務プロダクト)を中央の 1 ノードとして置き、その説明と内製/外部の境界を §プロダクト概要から引く。
- **網羅**: 直接利用アクター(`ACT-1`〜`ACT-6`)が漏れなく登場し、外部システム一覧の `EXT-*` がすべて登場すること。

**登場要素表の各列**:

- **ID**: 人は `ACT-*`(アクター一覧)、外部システムは `EXT-*`(外部システム一覧)をそれぞれ流用する(本書は ID を定義しない)。対象システムは単一のため付番しない(`—`)。
- **種別**: `人` / `対象システム` / `外部システム`。
- **名称**: 図のノード名と一致させる(アクター名はアクター一覧、外部システム名は外部システム一覧の名称を流用)。
- **説明・責務**: 人はアクター一覧の役割要約、外部システムは外部システム一覧の連携目的を引く。
- **参照元**: `ACT-*` は `アクター一覧`、`EXT-*` は `外部システム一覧`。

## 使用するテンプレート

出力時は以下のテンプレートを読み込み、その章構成・書き方ヒントに従う。

| テンプレートのパス | 内容 |
|---|---|
| `.claude/skills/d1-common-requirement-document/template.md` | ドメイン共通要求仕様書テンプレート(章構成・書き方ヒント) |

## 章構成

本書は以下の章で構成される。各節の要求は ID(`DOM-COMMON-<カテゴリ>-<連番>` 形式)を付与し、下流のドメイン要求仕様書群(個別)から「継承する共通要求」として参照される。

- §本書について(概要 / 想定読者 / 注記)
- §機能要求(DOM-COMMON-FR-*)
- §非機能要求(DOM-COMMON-NFR-*)
- §外部連携要求(DOM-COMMON-EXT-*)
- §UI/UX 方針(ID なし、観点ベースの方針宣言)
- §セキュリティ要求(DOM-COMMON-SEC-*)
  - §データアクセス要求(DOM-COMMON-SEC-DATA-*)
- §法務・コンプライアンス要求(DOM-COMMON-REG-*)
- §受け入れ基準(本書全体としての検収観点)

## 新規作成時の振る舞い

新規作成時(`docs/D1.business-requirements/common-requirement-document.md` 未存在)は **ユーザーへのヒアリングを行わず**、上流(PRD・BRD・アクター一覧・ドメイン定義書)と業界一般知識から本書を直接生成する。

1. PRD §「体験設計」・§「制約事項」・§「受け入れ基準」 と BRD §「事業戦略」「KPI」「セキュリティ要件」「法令・規制要件」 を読み、横断的に効く具体要求を抽出する
2. テンプレートの章順に従って書き出す。各章の主な引き取り元と注意点は以下:
    - **本書について**: 概要 / 想定読者 / 注記 の3つのサブ見出しに分割。「PRD は方針 / 本書は具体要求(共通)レイヤー / ドメイン要求仕様書群は具体要求(個別)レイヤー」との上下流関係を明示する
    - **機能要求**: 業務通知・帳票出力・業務の中断/再開・同意取得 等、複数ドメインで共通利用する **業務機能** を集約。認証・認可・監査ログ等のセキュリティの仕組みは §セキュリティ要求 へ、エラー検知・リトライ・冪等性等の信頼性は §非機能要求(信頼性) へ、外部サービス疎結合連携等のアーキ方針は §外部連携要求/§非機能要求 へ振り分ける
    - **非機能要求**: 性能(レスポンスタイム・スループット)・可用性・拡張性・運用性・保守性・互換性・信頼性 のカテゴリで整理
    - **外部連携要求**: 既存内部システム(契約管理・収納・帳票発送・商品マスタ管理・募集人管理)・外部サービス(本人確認・反社チェック・電子署名・決済) との連携を ID 付きで整理
    - **UI/UX 方針**: 対象デバイス・OS・ブラウザ / 利用シーン / アクセシビリティ / デザインシステム / 言語・通貨・地域 / エラー時の体験 等の観点で方針宣言。本節は ID を付けない方針集
    - **セキュリティ要求**: 業界基準準拠・認証・データ保護・アクセス制御・監査ログ・脆弱性対策・インシデント対応 等を集約。配下に §データアクセス要求(エンティティ×機密区分×アクセス制御) を入れ子で配置
    - **法務・コンプライアンス要求**: 適用法令・規制を列挙し、本書での求める対応・主たる影響範囲を整理
    - **受け入れ基準**: 本書記載の共通要求が満たされていることの検収観点を 3-7 項目で示す
3. ID は **`DOM-COMMON-<カテゴリ>-<連番>` 形式**。カテゴリは 機能要求=`FR` / 非機能要求=`NFR` / 外部連携要求=`EXT` / セキュリティ要求=`SEC` / 法務・コンプライアンス要求=`REG`。**データアクセス要求はセキュリティ要求の子のため、入れ子カテゴリ `SEC-DATA` をハイフン連結した `DOM-COMMON-SEC-DATA-<連番>`** とする
4. **ユーザー識別子は別途本書で作らず、アクター一覧の対応アクターID(`ACT-1`, `ACT-2`, ...)をそのまま用いる**(二重識別子を避ける)。複数アクターをまとめる場合は中黒区切り(例: `ACT-5・ACT-6`)
5. 上流から確定できない値には要確認マーカーを付ける(例: 具体的な性能数値・SLAの厳密値・セキュリティ準拠基準のバージョン 等)

## 留意事項

- **本書の規律(共通要求のみを書く)**: 「**3 ドメイン以上で共通して効く要求**」のみ本書に書く。2 ドメイン以下にしか効かない要求はドメイン要求仕様書(個別)に降ろす。本書を everything doc 化させない
- **§機能要求 は業務機能に絞る(がっつり業務寄せ)**: §機能要求 には「プロダクトが業務として提供する横断機能」(業務通知・帳票出力・業務の中断/再開・同意取得 等)のみ置く。認証・認可・監査ログ等のセキュリティの仕組みは §セキュリティ要求、エラー検知・リトライ・冪等性等の信頼性は §非機能要求(信頼性)、外部サービス疎結合連携等のアーキ方針は §外部連携要求/§非機能要求 に振り分ける。判定軸は「重複の有無」ではなく「**業務の話か**」で、セキュリティ・法務と多少重なっても業務機能なら機能要求に残してよい
- **下流成果物との棲み分け**: 以下は本書に書かず、各下流成果物に降ろす
    - 個別ドメインの業務ルール・業務状態遷移・業務運用(イレギュラー対応) → ドメイン要求仕様書群(個別)
    - ユーザーストーリー(誰が何のために) → ユーザーストーリー一覧
    - ユースケース(具体的な操作シナリオ) → ユースケース一覧
    - 機能の網羅リスト → 機能一覧(D2)
    - 画面の具体仕様 → 画面要件群(D2)
- **業務 + プロダクト観点に留まる**: 機能名・画面名・テーブル名・API 名等のシステム実装語は混入させない
- **What と How の境界(具体水準は本書で確定)**: 原則として How(具体的な実装手段)には踏み込まないが、**ビジネス・規制上譲れない具体水準**(可用性目標値・監査ログ保持期間・準拠すべき業界基準・暗号化アルゴリズム強度 等)は本書で確定する。判定軸は「**ビジネス・規制の非交渉性**」: 実装選択肢を狭めるだけの規定なら D2 以降に降ろし、ビジネス上の必達条件であれば本書に書く。「可用性は高水準を保つ」のようなフワッとした抽象化で逃げない
- **継承の可視化を意識**: 本書の各 ID(`DOM-COMMON-FR-1`, `DOM-COMMON-NFR-1`, `DOM-COMMON-SEC-1`, `DOM-COMMON-REG-1` 等)は後続のドメイン要求仕様書から「継承する共通要求」として参照される前提で設計する。`DOM-COMMON-` 名前空間により参照元成果物が一目で分かる。文言・粒度を一貫させ、表記揺れを起こさない
- **PRD との役割分担**: 方針・概念・体験設計・制約事項・受け入れ基準 は PRD の責務。本書は具体要求(共通)に責務を限定する。PRD と本書の二重定義を避ける
- **フェーズ2 ラベルの付与**: 初期フェーズの対象外で将来追加される要件(チャネル拡張・新商品・機械学習活用・将来連携 等)は `[フェーズ2]` ラベル(または「フェーズ2以降」記号)で明示し、初期/将来の境界を読み手に分からせる。BRD・PRD の同種ラベルと表記を揃える
- **上流との整合性**: BRDのスコープ・KPI・制約・フェーズ2 / アクター一覧の対象ユーザー / ドメイン定義書のドメイン構成 / PRD の方針 と矛盾させない

## 完成時のセルフチェック

執筆完了時に以下の観点で出力を検査し、不備があれば修正したうえで出力を完了とする。

- 本書が「3 ドメイン以上で共通する要求」のみに絞られており、everything doc 化していないか(2 ドメイン以下のものが混入していないか)
- §機能要求 が業務機能(業務通知・帳票出力・業務の中断/再開・同意取得 等)に絞られ、認証・認可・監査ログ(→セキュリティ)/ エラー検知・リトライ・冪等性(→非機能の信頼性)/ 外部サービス疎結合 等のアーキ方針 が混入していないか
- 個別ドメインの業務ルール・業務状態遷移・業務運用(イレギュラー対応)が混入していないか
- ユーザーストーリー・ユースケース・機能の網羅リスト・画面の具体仕様が混入していないか
- BRDのスコープ・KPI・制約と整合しているか(フェーズ2の境界も含む)
- 各節の要求表に ID(`DOM-COMMON-<カテゴリ>-<連番>` 形式)が付与され、後続のドメイン要求仕様書群(個別)から参照可能になっているか
- 機能名・画面名・テーブル名・API 名等のシステム実装語が混入していないか
- ビジネス・規制上譲れない具体水準(可用性目標値・ログ保持期間・準拠基準 等)が抽象化されすぎていないか(「可用性は高水準を保つ」等のフワッとした記述で逃げていないか)
- 将来要件・将来拡張に該当する箇所に `[フェーズ2]` ラベル(または同等表記)が付与されているか
- §UI/UX 方針 が方針宣言に留まり、具体画面仕様が混入していないか
- `{{` や XXX のプレースホルダが残っていないか

## 下流への伝搬関係

作成・更新時は以下の下流成果物との整合を意識する:

| 下流成果物 | ドメイン共通要求仕様書から何を引く |
|---|---|
| ドメイン要求仕様書群(個別) | 各節の `DOM-COMMON-*` を「継承する共通要求」として参照 |
| ユーザーストーリー一覧 | 横断ストーリーの上流参照(`紐づくドメイン共通要求 ID` 列) |
| 画面コンセプト集 | 体験原則の上流参照(`DOM-COMMON-*`) |
| ユースケース一覧 / 機能一覧(D2) / 画面要件群(D2) | 横断機能・横断要件の起点として参照 |

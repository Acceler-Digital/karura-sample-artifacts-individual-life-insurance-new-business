## 新規作成時の振る舞い

新規作成時(`docs/D1.business-requirements/domain-definition-document.md` 未存在)は **ユーザーへのヒアリングを行わず**、上流の BRD と業界一般知識からドメインを抽出して直接生成する。

1. BRD(`docs/D1.business-requirements/business-requirement-document.md`)を読み込み、対象業務・スコープから **業務工程ドメイン候補** を抽出する。BRD の「スコープ」表に列挙された業務単位を起点にしつつ、**スコープ表に明示されていない準備工程**(顧客との初期対話・意向把握・要件聴取 等の最上流工程)および **事後業務** が業務として存在しないかを業界一般知識から検証する。BRD のスコープ表の粒度がそのままドメイン粒度になるとは限らない点に注意する
2. BRD・業界知識から、業務工程をまたがって参照される **横断ドメイン候補** を抽出する(顧客情報・商品マスタ・本人確認・反社チェック・電子署名 等)。「将来のビジネスの拡張性を担保するためのシステム的要求」セクションに記載の領域(マルチチャネル対応・データ活用基盤連携 等)も拾い上げる。**規制対応系の領域**(意向把握・情報提供・適合性原則・本人確認・告知 等)を扱うときは、「**業務プロセスそのもの**」(業務工程ドメインとして配置)と「**証跡の収集・保全**」(横断ドメインとして配置)に分けて捉え、両者を同一ドメインに混在させない
3. 各ドメインを2軸で分類する:
    - **区分**: コア / サポート / 汎用(プロダクトの差別化に直結するか、業務上不可欠か、業界横断で汎用化されているか)
    - **種別**: 業務工程ドメイン / 横断ドメイン
4. テンプレート(`.claude/skills/d1-domain-definition-document/template.md`)の形式に従って書き出す。BRD・業界知識から確定できない値には要確認マーカーを付ける
5. 「ドメイン間の主な連携」(Mermaid 図)は、業務工程の前後関係(実線矢印)と横断ドメインへの参照関係(破線矢印)を主要な関係に絞って描く。BRD のスコープ表で業務フローが十分に表現されており、横断ドメインへの参照も「概要」列で読み取れる場合は省略してよい

## 留意事項

- **問題空間に留まる**: 機能名・画面名・テーブル名・サービス名・API 名などは混入させない。これらは下流の PRD・ドメイン要求仕様書群・C4 モデル等に降ろす
- **MECE性**: BRD のスコープを過不足なくカバーする。業務工程ドメインだけでなく横断ドメインの取りこぼしに注意する。BRD スコープ表の業務単位粒度がそのままドメイン粒度になるとは限らないため、**準備工程・事後業務の取りこぼし**、および **規制対応領域における業務プロセス/証跡管理の混在** に特に注意する
- **Bounded Context との混同を避ける**: 本書は問題空間の整理であり、「1ドメイン=1サービス」「1ドメイン=1Bounded Context」を前提として設計してはならない。マッピングは下流フェーズで決定する
- **BRD との整合性**: 上流の BRD のスコープ・対象業務と整合させる

## 完成時のセルフチェック

執筆完了時に以下の観点で出力を検査し、不備があれば修正したうえで出力を完了とする。

- BRD のスコープを過不足なくカバーしているか(MECE に分解できているか)
- BRD スコープ表に明示されていない準備工程(初期対話・意向把握 等)・事後業務の取りこぼしがないか
- 業務工程ドメインだけでなく、横断ドメインも挙げられているか
- 規制対応系の領域で「業務プロセス」と「証跡管理」が同一ドメインに混在していないか(同一ドメインに2つ以上の異なる性質の責務を混在させない)
- 機能・画面・テーブル名・サービス名が混入していないか(問題空間に留まっているか)
- 各ドメインの区分(コア/サポート/汎用)がプロダクト戦略と整合しているか
- `{{` や XXX のプレースホルダが残っていないか

## 下流への伝搬関係

作成・更新時は以下の下流成果物との整合を意識する:

| 下流成果物 | ドメイン定義書から何を引く |
|---|---|
| プロダクト要求仕様書(PRD) | スコープのドメイン分割の根拠、ドメインごとの戦略上の位置づけ(コア/サポート/汎用) |
| ドメイン要求仕様書群 | 各冊の構成単位(1ドメイン=1つ) |

| 成果物のパス | 内容 |

## 新規作成時の振る舞い

新規作成時(`docs/D1.business-requirements/product-requirement-document.md` 未存在)は **ユーザーへのヒアリングを行わず**、上流3成果物(BRD・アクター一覧・ドメイン定義書)と業界一般知識から PRD を直接生成する。

1. 上流3成果物を読み込み、本書の各章にどの成果物の何を引き取るかを整理する
2. テンプレートの章順に従って書き出す。各章の主な引き取り元と注意点は以下:
    - **本書について**: 概要 / 想定読者 / 注記 の3つのサブ見出しに分割。概要では「BRD は Why + 事業レベル What を扱う」「PRD はそれらを Why として引き継ぎプロダクトレベル What を扱う」「本書は親文書としてドメイン要求仕様書群へ共通要求を継承させる」を明示する
    - **プロダクト概要**: BRD「事業戦略とプロダクト概要」をプロダクト視点で言い換え(業務フローではなく「何を提供するか」)。将来拡張(チャネル拡張・新商品・機械学習活用 等)は `[フェーズ2]` ラベルで明示
    - **体験設計 > 提供価値・体験方針**: BRD「事業戦略」「KPI」「マーケティング戦略」「将来拡張性」から直接派生する設計原則。3〜6柱に絞る。利用ユーザーへの絞り込みは次節に委ね、本節では行わない
    - **体験設計 > システム利用ユーザーと体験設計**: アクター一覧から直接利用アクターのみを抽出したフィルタビュー。アクター一覧の役割・責務・関心事を本節で重複記載せず、PRD レベルで体験設計に効く観点(配慮点・利用形態)に絞る。配下に「システム利用ユーザーと体験設計上の配慮点」表と「利用形態」表を配置
    - **ドメイン構成と要求仕様書群への分割方針**: ドメイン定義書の各ドメインに対して対応するドメイン要求仕様書を割り当てる(原則1ドメイン=1要求仕様書)
    - **プロダクト横断の要求事項**: 親章「プロダクト横断の要求事項」の配下に 機能要求・非機能要求・外部連携要求・UI/UX 方針・セキュリティ要求・法務/コンプライアンス要求 の各節(`###`)を置く。**データアクセス要求(エンティティ×機密区分×アクセス制御)は「データをどう守るか」がアクセス制御中心のため、独立節にせずセキュリティ要求の子(`####`)とする**(機密区分は親セキュリティ統制の入力であり、法務・D2データモデルからも参照される)。「3ドメイン以上で共通して効く要求のみ/個別ドメイン固有は降ろす/後続から継承される」という親PRDの規律は親章の導入文に集約し、各節導入文では繰り返さない。各節は BRDの該当章および将来拡張性の章を機能・運用要求として翻訳。将来要件は `[フェーズ2]` ラベルで初期/将来の境界を明示。**制約事項・受け入れ基準 は「要求事項」ではないため本章に含めず、独立章(`##`)とする**
    - **制約事項**: BRDのスケジュール・予算・人事的制約のうちPRD段階で意識すべきものを抜粋
    - **受け入れ基準**: BRDの「プロジェクトの成功の定義」とKPIから派生
3. 上流から確定できない値には要確認マーカーを付ける(例: 具体的な性能数値・SLAの厳密値・セキュリティ準拠基準のバージョン 等)
4. ID は **`PRD-<カテゴリ>-<連番>` 形式**(成果物名 `PRD` を名前空間とし、PRD が振った ID であることを明示する)。カテゴリは 機能要求=`FR` / 非機能要求=`NFR` / 外部連携要求=`EXT` / セキュリティ要求=`SEC` / 法務・コンプライアンス要求=`REG`。**データアクセス要求はセキュリティ要求の子のため、入れ子カテゴリ `SEC-DATA` をハイフン連結した `PRD-SEC-DATA-<連番>`** とする。例: `PRD-FR-1`, `PRD-NFR-1`, `PRD-EXT-1`, `PRD-SEC-1`, `PRD-SEC-DATA-1`, `PRD-REG-1`。**ユーザー識別子は別途PRDで作らず、アクター一覧の対応アクターID(`ACT-1`, `ACT-2`, ...)をそのまま用いる**(二重識別子を避ける)。複数アクターをまとめる場合は中黒区切り(例: `ACT-5・ACT-6`)
5. §「体験設計 > システム利用ユーザーと体験設計」配下の表は **アクター名を先頭、アクターIDを2列目** に並べる(ユーザー名が読み手の関心事、IDは追跡用の補助)。配慮点列は `<br>` + `・` の改行記法で箇条化して視認性を上げる

## 留意事項

- **親PRDの規律(共通要求のみを書く)**: 「**3ドメイン以上で共通して効く要求**」のみ本書に書く。2ドメイン以下にしか効かない要求はドメイン要求仕様書に降ろす。本書を everything doc 化させない
- **機能要求は業務機能に絞る(がっつり業務寄せ)**: §機能要求 には「プロダクトが業務として提供する横断機能」(業務通知・帳票出力・業務の中断/再開・同意取得 等)のみ置く。認証・認可・監査ログ 等のセキュリティの仕組みは §セキュリティ要求、エラー検知・リトライ・冪等性 等の信頼性は §非機能要求(信頼性)、外部サービス疎結合連携 等のアーキ方針は §外部連携要求/§非機能要求 に振り分ける。判定軸は「重複の有無」ではなく「**業務の話か**」で、セキュリティ・法務と多少重なっても業務機能なら機能要求に残してよい
- **下流成果物との棲み分け**: 以下は本書に書かず、各下流成果物に降ろす
    - 個別ドメインの業務ルール・業務状態遷移・業務運用(イレギュラー対応) → ドメイン要求仕様書群
    - ユーザーストーリー(誰が何のために) → ユーザーストーリー一覧
    - ユースケース(具体的な操作シナリオ) → ユースケース一覧
    - 機能の網羅リスト → 機能一覧(D2)
    - 画面の具体仕様 → 画面要件群(D2)
- **業務 + プロダクト観点に留まる**: 機能名・画面名・テーブル名・API 名等のシステム実装語は混入させない
- **What と How の境界(具体水準は本書で確定)**: 原則として How(具体的な実装手段)には踏み込まないが、**ビジネス・規制上譲れない具体水準**(可用性目標値・監査ログ保持期間・準拠すべき業界基準・暗号化アルゴリズム強度 等)は本書で確定する。判定軸は「**ビジネス・規制の非交渉性**」: 実装選択肢を狭めるだけの規定なら D2 以降に降ろし、ビジネス上の必達条件であれば本書に書く。「可用性は高水準を保つ」のようなフワッとした抽象化で逃げない
- **継承の可視化を意識**: 本書の各 ID(`PRD-FR-1`, `PRD-NFR-1`, `PRD-SEC-1`, `PRD-REG-1` 等)は後続のドメイン要求仕様書から「継承する PRD §」として参照される前提で設計する。`PRD-` 名前空間により参照元成果物が一目で分かる。文言・粒度を一貫させ、表記揺れを起こさない
- **アクター一覧との関係(フィルタビュー)**: 本書の「システム利用ユーザーと体験設計」節は、アクター一覧で網羅された全アクター(直接利用・業務関係者・統制/監督)のうち **本プロダクトを直接利用するアクターのみを抽出したフィルタビュー** であり、役割・責務・関心事はアクター一覧を正典とする。本節は PRD 固有レンズ(配慮点・利用形態)のみを書き、アクター一覧の情報を重複記載しない
- **章順の規律(BRD → 何を → 誰に)**: §「体験設計」は「提供価値・体験方針 → システム利用ユーザーと体験設計」の順序を守る。提供価値は BRD から直接派生する設計原則であり、利用ユーザーは「その価値を届ける対象の絞り込み」として後続するため、逆順にしない
- **フェーズ2 ラベルの付与**: 初期フェーズの対象外で将来追加される要件(チャネル拡張・新商品・機械学習活用・将来連携 等)は `[フェーズ2]` ラベル(または「フェーズ2以降」記号)で明示し、初期/将来の境界を読み手に分からせる。BRD の同種ラベルと表記を揃える
- **上流との整合性**: BRDのスコープ・KPI・制約・フェーズ2 / アクター一覧の対象ユーザー / ドメイン定義書のドメイン構成 と矛盾させない

## 完成時のセルフチェック

執筆完了時に以下の観点で出力を検査し、不備があれば修正したうえで出力を完了とする。

- 本書が「ドメイン横断・共通要求」に絞られており、everything doc 化していないか
- §機能要求 が業務機能(業務通知・帳票出力・業務の中断/再開・同意取得 等)に絞られ、認証・認可・監査ログ(→セキュリティ)/ エラー検知・リトライ・冪等性(→非機能の信頼性)/ 外部サービス疎結合 等のアーキ方針 が混入していないか
- 3ドメイン以上で共通する要求のみが本書に置かれているか(2ドメイン以下のものが混入していないか)
- 個別ドメインの業務ルール・業務状態遷移・業務運用(イレギュラー対応)が混入していないか
- ユーザーストーリー・ユースケース・機能の網羅リスト・画面の具体仕様が混入していないか
- BRDのスコープ・KPI・制約と整合しているか(フェーズ2の境界も含む)
- §「体験設計 > システム利用ユーザーと体験設計」がアクター一覧のフィルタビューとして機能し、役割・責務・関心事の重複記載がないか
- §「体験設計」配下の章順が「提供価値・体験方針 → システム利用ユーザーと体験設計」になっているか(BRD → 何を → 誰に の派生順)
- §「体験設計 > システム利用ユーザーと体験設計」配下の表で **アクター名を先頭、対応アクターID(`ACT-n`) を2列目** に置き、PRD 独自の `USR1` 等の二重識別子が混入していないか
- 配慮点セルが `<br>` + `・` の改行記法で箇条化されており、視認性が確保されているか
- ドメイン定義書の **全ドメイン** が「ドメイン構成と要求仕様書群への分割方針」章で漏れなく扱われているか
- §「プロダクト横断の要求事項」配下に 機能要求・非機能要求・外部連携要求・UI/UX 方針・セキュリティ要求(子に `####` データアクセス要求)・法務/コンプライアンス要求 が漏れなく置かれ、各要求表に ID(`PRD-<カテゴリ>-<連番>` 形式)が付与されて後続のドメイン要求仕様書から参照可能になっているか
- 制約事項・受け入れ基準 が「プロダクト横断の要求事項」配下に誤って取り込まれず、独立章(`##`)として保たれているか
- 機能名・画面名・テーブル名・API 名等のシステム実装語が混入していないか
- ビジネス・規制上譲れない具体水準(可用性目標値・ログ保持期間・準拠基準 等)が抽象化されすぎていないか(「可用性は高水準を保つ」等のフワッとした記述で逃げていないか)
- 将来要件・将来拡張に該当する箇所に `[フェーズ2]` ラベル(または同等表記)が付与されているか
- `{{` や XXX のプレースホルダが残っていないか

## 下流への伝搬関係

作成・更新時は以下の下流成果物との整合を意識する:

| 下流成果物 | PRDから何を引く |
|---|---|
| ドメイン要求仕様書群 | §「ドメイン構成と要求仕様書群への分割方針」/ §「プロダクト横断の要求事項」配下の各節を「継承するPRD §」として参照 |
| ユーザーストーリー一覧 | §「体験設計 > システム利用ユーザーと体験設計」/ §「体験設計 > 提供価値・体験方針」 |
| ユースケース一覧 | §「体験設計 > システム利用ユーザーと体験設計」/ §「体験設計 > 提供価値・体験方針」(ユーザーストーリー一覧からさらに派生) |
| 機能一覧(D2) | §「プロダクト横断の要求事項 > 機能要求」 + 各ドメイン要求仕様書から導出される機能 |
| 画面要件群(D2) | §「UI/UX方針」 + §「体験設計 > 利用形態」 + 各ドメイン要求仕様書 + 機能一覧 |

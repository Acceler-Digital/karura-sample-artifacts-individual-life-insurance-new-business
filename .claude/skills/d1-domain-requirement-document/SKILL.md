## 章構成

本書は専用の「対象ドメイン」「継承するPRD要求」「他ドメインとの連携」「想定読者」章を **持たない**。ドメイン定義は正典であるドメイン定義書を参照し、PRD横断要求の継承・他ドメイン連携は各業務ルールの「根拠/制約」列および「受け入れ基準」にインラインで紐づける(専用の継承章を設けると PRD と二重管理になりドリフトするため)。固定章構成は以下:

```
# {{ドメイン名}}要求仕様書
## 本書について
### 概要        … 定型2文(ドメイン名のみ置換)
### 注記        … 定型1文
## 業務要求
### 業務ルール
### 業務状態遷移
### 業務運用(イレギュラー対応)
## セキュリティ要求
### データアクセス要求
## 受け入れ基準
```

## 新規作成時の振る舞い

新規作成時は **ユーザーへのヒアリングを行わず**、PRD・ドメイン定義書・アクター一覧・BRD と業界一般知識から各ドメイン要求仕様書を直接生成する。1ドメインずつ、テンプレートの章順に従って書き出す。

1. **本書について**: テンプレートの定型文をそのまま用い、`{{ドメイン名}}` をドメイン定義書「一覧」の名と一致させて置換する。概要・注記以外の前文は設けない
2. **業務ルール**: BRD のスコープ・法令制約、ドメイン定義書の「主な関心事」「ドメイン間の主な連携」、業界一般知識から、本ドメイン固有の判断基準・制約・条件を導く。PRD で既に共通化されている横断要求は再掲せず、本ドメインに効くものを「根拠/制約」列に PRD ID(`PRD-FR-1` 等)で紐づける。他ドメインとの連携・ドメイン定義書の関心事も根拠列に明記し、専用章を持たない分のトレーサビリティをここで担保する。本ドメインの**責務境界**(他ドメイン・既存システムとの分担)が明確な場合は、その範囲を業務ルールとして1項立てる(例: 計上完了までを担い継続収納は既存収納システムに引き渡す、本ドメインは証跡の収集・保全に責務を限定し業務判断ロジックは工程ドメインに委ねる 等)
3. **業務状態遷移**: 本ドメインが管理する主要業務対象の業務状態と遷移(状態定義表+遷移表)を示す。導入文では**主要業務対象を「」付きで明示**する(例: 「主要な業務対象である『申込』の業務状態と遷移を示します」)。独立した複数の業務対象を持つドメインは Mermaid の composite state(`state "名前" as エイリアス { ... }`)で1つの図に並置する。遷移ラベルに `(XXX-BR-N)` 形式の業務ルールIDを括弧書きで添えると図と表のトレーサビリティが上がる
4. **業務運用(イレギュラー対応)**: 正常系から外れる業務局面と業務上の取り扱いを示す。「業務上の対応」列の末尾に `(XXX-BR-N に整合)` / `(PRD-NFR-9 に整合)` 形式で関連IDを併記し、対応の根拠と整合先を可視化する
5. **セキュリティ要求 > データアクセス要求**: PRD §「セキュリティ要求 > データアクセス要求」の機密区分を継承し、本ドメイン固有の主要データを対応づける。構造詳細は D2 に降ろす。表の「PRD 機密区分との対応」列でPRD継承が示されるため、導入文(「PRD §〜の機密区分を継承し、本ドメイン固有の主要データを以下に対応づけます。」)は**省略可**。PRD継承の文脈を明示したい場合のみ残す
6. **受け入れ基準**: 継承 PRD 要求の本ドメインでの充足・業務ルールの網羅・主要業務状態遷移の通し確認 等を3〜7項目で示す。専用の継承章を持たないため、継承 PRD 要求の充足確認は以下のいずれかで担保する: (A) 「継承 PRD 要求の充足: ...」として独立した1項目を立てる(汎用)、(B) 各ドメイン固有項目の末尾に `(PRD-REG-1 / PRD-REG-6 充足)` 形式でインラインPRD ID併記(規制対応色が強く項目とPRDが1対1で対応するドメイン向け)。形式 A と B は併用可。いずれの形式でも、本書が継承するPRD要求が受け入れ基準のどこかで漏れなく充足確認されること
7. 上流から確定できない値には要確認マーカーを付ける。将来段階導入の要求には `[フェーズ2]` ラベルを付ける
8. ID は **`<ドメインコード>-<カテゴリ>-<連番>` 形式**(成果物略号=当該ドメインコード)。カテゴリは 業務ルール=`BR` / 業務運用(イレギュラー)=`IRR` / ドメイン固有データ=`DATA`。例: `HEAR-BR-1`, `HEAR-IRR-1`, `HEAR-DATA-1`。CLAUDE.md §8 に準拠。継承する PRD 要求は PRD ID(`PRD-FR-1` 等)をそのまま用い、再採番しない

## 留意事項

- **継承は根拠列でインライン可視化**: 横断要求は PRD を正典とする。専用の継承章は持たず、本書は各業務ルールの「根拠/制約」列に PRD ID をそのまま紐づけ、内容は再掲しない。PRD で共通化済みの要求を本書で重複定義すると二重管理になりドリフトする
- **ドメイン定義書の定義を改変しない**: 対象ドメインの ID・名・区分・種別・概要・主な関心事 はドメイン定義書が正典。専用の「対象ドメイン」章は持たないが、業務ルール・データ要求の根拠はドメイン定義書の関心事・連携と矛盾させない(齟齬を見つけたらドメイン定義書側の修正を促す)
- **ドメインの What に留まる**: 「本ドメインとして何を満たすべきか」を業務言語で書く。機能名・画面名・テーブル名・API 名等の How(D2 以降)は混入させない。ただしビジネス・規制上譲れない本ドメイン固有の具体水準は本書で確定する
- **システム寄り用語に「業務」を冠する**: 状態遷移→**業務状態遷移**、例外処理→**業務運用(イレギュラー対応)** のように、D1 成果物ではシステム実装を想起させる語に「業務」を明示し、業務観点であることを保つ
- **業務ルールが中核**: 本書の価値は当該ドメイン固有の業務ルール・判断基準・条件分岐の網羅にある。各ルールに根拠(法令 / PRD §ID / ドメイン定義書の関心事・連携)を必ず紐づける。専用の継承章・他ドメイン連携章を持たない分、トレーサビリティはこの根拠列が担う
- **責務境界を業務ルール化する**: 後続ドメイン・既存システムとの分担が明確なドメインでは、「本ドメインの責務はXXXまでとし、以降は……の責務とする」を業務ルールとして1項立てる。これにより下流の機能一覧・C4・シーケンス図で責務漏れ/重複を起こさない
- **連携はドメイン定義書を起点**: 専用の「他ドメインとの連携」章は持たないが、業務ルール・状態遷移の根拠/契機はドメイン定義書の「ドメイン間の主な連携」と矛盾させない。詳細シーケンスは下流の L1/L2 シーケンス図に降ろす
- **フェーズ2 ラベル**: 初期フェーズ対象外の業務ルール・データは `[フェーズ2]` ラベルで境界を明示し、PRD・BRD の同種ラベルと表記を揃える
- **1ドメイン=1ファイル**: ドメイン定義書の全ドメインに対し原則1ファイル。PRD「ドメイン構成と要求仕様書群への分割方針」での統合/分割判断に従う

## 完成時のセルフチェック

執筆完了時に以下の観点で出力を検査し、不備があれば修正したうえで出力を完了とする。

- 固定章構成(本書について[概要/注記]→業務要求[業務ルール/業務状態遷移/業務運用]→セキュリティ要求[データアクセス要求]→受け入れ基準)に一致しているか。廃止章(対象ドメイン/継承するPRD要求/他ドメインとの連携/想定読者)を復活させていないか
- PRD の適用範囲列に当該ドメインが含まれる横断要求を、漏れなく業務ルールの「根拠/制約」列に PRD ID で紐づけているか(継承内容を再掲して二重管理になっていないか)
- 業務ルール・データ要求の根拠がドメイン定義書の関心事・連携と整合しているか(改変・矛盾がないか)
- 本ドメイン固有の業務ルール・業務状態遷移・業務運用(イレギュラー対応)が網羅され、各業務ルールに根拠が紐づいているか
- 他ドメイン・既存システムとの責務境界が明確なドメインで、その境界が業務ルールとして明示されているか
- 業務状態遷移の導入文で**主要業務対象が「」付きで明示**されているか(汎用文「主要な業務対象の業務状態と遷移を示します」のままになっていないか)
- 複数の独立業務対象を持つドメインで composite state による並置表現が検討されているか
- 業務運用(イレギュラー対応)の「業務上の対応」列に、関連 BR/IRR/PRD ID が `(XXX-BR-N に整合)` 形式で紐づけられているか
- 状態遷移・例外処理 等のシステム寄り用語に「業務」が冠されているか
- データアクセス要求が PRD §「セキュリティ要求 > データアクセス要求」の機密区分と整合しているか(導入文の有無は不問)
- 受け入れ基準に「継承 PRD 要求の充足」確認が、独立項目(形式A)またはインライン併記(形式B)のいずれかで担保されているか
- 機能名・画面名・テーブル名・API 名等の How(D2以降)が混入していないか
- 各表に ID(`<ドメインコード>-<カテゴリ>-<連番>` 形式)が付与され、後続成果物から参照可能になっているか
- 将来要件に `[フェーズ2]` ラベルが付与されているか
- `{{` や XXX のプレースホルダ、不要な要確認マーカーが残っていないか

## 下流への伝搬関係

作成・更新時は以下の下流成果物との整合を意識する:

| 下流成果物 | ドメイン要求仕様書から何を引く |
|---|---|
| ユーザーストーリー一覧 | 当該ドメインの業務ルール・業務運用(誰が何のために) |
| ユースケース一覧 | 業務状態遷移・業務運用(イレギュラー対応)から具体シナリオ |
| 機能一覧(D2) | 業務ルール + その根拠列が参照する継承 PRD 要求から導出される機能 |
| 画面要件群(D2) | 業務状態遷移・業務運用 + 機能一覧 |
| L1/L2 シーケンス図 | 業務状態遷移・業務ルールの根拠/契機が指すドメイン定義書の連携 |

## 新規作成時の振る舞い

新規作成時は **ユーザーへのヒアリングを行わず**、PRD・ドメイン定義書・アクター一覧・BRD と業界一般知識から各ドメイン要求仕様書を直接生成する。1ドメインずつ、テンプレートの章順に従って書き出す。

1. **対象ドメイン**: ドメイン定義書の「一覧」から該当ドメインの1行(ID・名・区分・種別・概要・主な関心事)をそのまま引き取る。ドメイン定義書が正典であり、本書で定義を改変しない
2. **継承するPRD要求**: PRD の各横断要求(PRD-FR / PRD-NFR / PRD-EXT / PRD-SEC / PRD-SEC-DATA / PRD-REG 等)の適用範囲列に当該ドメインが含まれるものを漏れなく拾い、PRD ID をそのまま参照する。実体は PRD が正典のため内容を再掲せず、「本ドメインでの適用観点」のみ補足する
3. **ドメイン固有の業務要求**: BRD のスコープ・法令制約、ドメイン定義書の「主な関心事」、業界一般知識から、本ドメイン固有の **業務ルール / 業務状態遷移 / 業務運用(イレギュラー対応)** を導く。PRD で既に共通化されている要求はここで重複定義しない
4. **他ドメインとの連携**: ドメイン定義書の「ドメイン間の主な連携」(業務工程の前後関係・横断ドメインへの参照)から本ドメインの入力・出力を整理する
5. **ドメイン固有のデータ要件**: PRD §「セキュリティ要件 > データアクセス要件」の機密区分を継承し、本ドメイン固有の主要データを対応づける。構造詳細は D2 に降ろす
6. **受け入れ基準**: 継承 PRD 要求の本ドメインでの充足・業務ルールの網羅・主要業務状態遷移の通し確認 等を3〜7項目で示す
7. 上流から確定できない値には要確認マーカーを付ける。将来段階導入の要求には `[フェーズ2]` ラベルを付ける
8. ID は **`<ドメインコード>-<カテゴリ>-<連番>` 形式**(成果物略号=当該ドメインコード)。カテゴリは 業務ルール=`BR` / 業務運用(イレギュラー)=`BOP` / ドメイン固有データ=`DATA`。例: `HEAR-BR-1`, `HEAR-BOP-1`, `HEAR-DATA-1`。CLAUDE.md §8 に準拠。継承する PRD 要求は PRD ID(`PRD-FR-1` 等)をそのまま用い、再採番しない

## 留意事項

- **親はPRD(継承の可視化)**: 横断要求は PRD を正典とする。本書は PRD ID をそのまま参照し、内容を再掲しない。PRD で共通化済みの要求を本書で重複定義すると二重管理になりドリフトする
- **ドメイン定義書の定義を改変しない**: 対象ドメインの ID・名・区分・種別・概要・主な関心事 はドメイン定義書が正典。本書で書き換えない(齟齬を見つけたらドメイン定義書側の修正を促す)
- **ドメインの What に留まる**: 「本ドメインとして何を満たすべきか」を業務言語で書く。機能名・画面名・テーブル名・API 名等の How(D2 以降)は混入させない。ただしビジネス・規制上譲れない本ドメイン固有の具体水準は本書で確定する
- **システム寄り用語に「業務」を冠する**: 状態遷移→**業務状態遷移**、例外処理→**業務運用(イレギュラー対応)** のように、D1 成果物ではシステム実装を想起させる語に「業務」を明示し、業務観点であることを保つ
- **業務ルールが中核**: 本書の価値は当該ドメイン固有の業務ルール・判断基準・条件分岐の網羅にある。各ルールに根拠(法令 / PRD §ID / ドメイン定義書の関心事)を必ず紐づける
- **連携はドメイン定義書を起点**: 他ドメインとの連携はドメイン定義書の「ドメイン間の主な連携」と矛盾させない。詳細シーケンスは下流の L1/L2 シーケンス図に降ろす
- **フェーズ2 ラベル**: 初期フェーズ対象外の業務ルール・連携・データは `[フェーズ2]` ラベルで境界を明示し、PRD・BRD の同種ラベルと表記を揃える
- **1ドメイン=1ファイル**: ドメイン定義書の全ドメインに対し原則1ファイル。PRD「ドメイン構成と要求仕様書群への分割方針」での統合/分割判断に従う

## 完成時のセルフチェック

執筆完了時に以下の観点で出力を検査し、不備があれば修正したうえで出力を完了とする。

- 対象ドメインの定義がドメイン定義書と一致しているか(改変していないか)
- PRD の適用範囲列に当該ドメインが含まれる横断要求を、漏れなく「継承する PRD §」として参照しているか
- 継承した PRD 要求の内容を再掲して二重管理になっていないか(参照+適用観点のみになっているか)
- 本ドメイン固有の業務ルール・業務状態遷移・業務運用(イレギュラー対応)が網羅され、各業務ルールに根拠が紐づいているか
- 状態遷移・例外処理 等のシステム寄り用語に「業務」が冠されているか
- 他ドメイン連携がドメイン定義書の連携と整合しているか
- データ要件が PRD §「セキュリティ要件 > データアクセス要件」の機密区分と整合しているか
- 機能名・画面名・テーブル名・API 名等の How(D2以降)が混入していないか
- 各表に ID(`<ドメインコード>-<カテゴリ>-<連番>` 形式)が付与され、後続成果物から参照可能になっているか
- 将来要件に `[フェーズ2]` ラベルが付与されているか
- `{{` や XXX のプレースホルダ、不要な要確認マーカーが残っていないか

## 下流への伝搬関係

作成・更新時は以下の下流成果物との整合を意識する:

| 下流成果物 | ドメイン要求仕様書から何を引く |
|---|---|
| ユーザーストーリー一覧 | 当該ドメインの業務ルール・業務運用(誰が何のために) |
| ユースケース一覧 | 業務状態遷移・業務運用(イレギュラー対応)から具体シナリオ |
| 機能一覧(D2) | 業務ルール + 継承 PRD 要求から導出される機能 |
| 画面要件群(D2) | 業務状態遷移・業務運用 + 機能一覧 |
| L1/L2 シーケンス図 | 他ドメインとの連携・業務状態遷移 |

## 章構成

本書は **「本書について」「想定読者」「注記」「受け入れ基準」 等の編集メタ章を持たない**。タイトル(`# ユースケース一覧`)直下に単一のフラット表のみが置かれる構成。前提・位置づけ・編集チェック観点は本スキル(および テンプレ冒頭の HTML コメントブロック)に集約し、成果物本体には残さない。スプレッドシート/Excel への移送を想定し、ドメイン別の章分割は行わない。

```
# ユースケース一覧

| ID | ドメイン / 区分 | UC 名 | 主アクター | トリガ | 主要価値 | 関連 US | 関連要求 ID |
|---|---|---|---|---|---|---|---|
| UC-1 | 意向把握 | 意向を把握する | 募集人: ACT-1・ACT-2 | ... | ... | US-1〜12 | DOM-HEAR-BR-1, ... |
| UC-2 | 横断     | 本プロダクトにログインする | 全直接利用アクター | ... | ... | US-100〜105 | DOM-COMMON-SEC-1, DOM-COMMON-SEC-2 |
```

**シーケンス図分割単位としての規律**: 後続の D2 システム要件・D3 システム設計でシーケンス図が描けるよう、各 UC は「主アクター 1(または同等業務責務の複数アクター)・トリガ 1・主要価値 1」が明確であること。1 UC で複数の独立した業務シナリオを兼任させない(兼任が必要なら UC を分割する)。

## 新規作成時の振る舞い

新規作成時(`docs/D1.business-requirements/use-case-list.md` 未存在)は **ユーザーへのヒアリングを行わず**、上流(ユーザーストーリー一覧・ドメイン要求仕様書群・PRD・アクター一覧)から本書を直接生成する。

1. ユーザーストーリー一覧を読み込み、全 US を業務局面 × 主アクターで仮グループ化する
2. 仮グループを「ユーザーにとってビジネス上価値がある単位」に整える:
    - **シーケンス図が独立して描ける粒度** であること(トリガ・主アクター・成果が明確)
    - **同一業務責務の複数アクター(募集人 ACT-1・ACT-2 等)は 1 UC に統合** する(主アクター列に中黒区切りで併記)
    - **業務責務が異なるアクターは別 UC** とする(例: 申込受付 = 募集人 / 停留申込収束 = 新契約事務担当者)
    - 1 ドメイン = 1 UC に固執しない。1 ドメイン内に業務局面が複数あれば複数 UC に分割してよい(例: 引受査定 / 自動判定基準調整)
3. 横断 US(`区分=横断`)の扱い:
    - **認証(ログイン)・業務の中断/再開 等、それ自体が独立した業務局面となる横断ストーリーは独立 UC** として立てる(全直接利用アクターを 1 UC にまとめる、または被保険者・募集人の区分が違う場合は分割)
    - **通知・帳票出力・同意取得/撤回・外部連携障害時縮退・例外UX 等、業務局面 UC のサブシナリオに過ぎない横断ストーリーは独立 UC を立てず、当該業務局面 UC の「関連 US」列に取り込む**(代替シナリオ・前提条件として包含)
4. 各 UC は目標 15-25 件に収める。これを大きく超える場合は集約不足、下回る場合は業務局面の取りこぼしの可能性を点検する
5. 各列の埋め方:
    - **ID**: `UC-<連番>` 形式の通し番号(例: `UC-1`, `UC-2` …)。ドメイン情報は ID に含めず「ドメイン / 区分」列で表現する
    - **ドメイン / 区分**: ドメイン名(例: `意向把握`)または `横断`。複数ドメインにまたがる場合は主たるドメイン名を記載(横断は §横断の判定基準に従う)
    - **UC 名**: 「{{動詞}}する」または「{{目的語}}を{{動詞}}する」の能動形(例: `意向を把握する` / `申込を受け付ける` / `本プロダクトにログインする`)。システム実装語(画面名・API 名・テーブル名)は混入させない
    - **主アクター**: 当該 UC を駆動する主たるアクター。同一業務責務の複数アクターは中黒区切り(例: `募集人: ACT-1・ACT-2`)。被保険者本人・申込人本人など本人系も同様に併記可。**業務責務が異なるアクターは別 UC に分割** する
    - **トリガ**: UC を開始する業務イベント(例: `顧客との初回面談』, `第一回保険料の支払期日到来`, `契約成立確定`)。業務言語で記述
    - **主要価値**: ユーザーにとってのビジネス上の価値・このUC が必要な業務根拠(例: `適合性原則違反のリスクなく後工程へ意向を引き渡せる`)。なぜ必要かを業務観点で記述。KPI への直接効果があれば BRD KPI を引用
    - **関連 US**: 当該 UC に紐づくユーザーストーリーの ID。連続範囲は `US-1〜12` のように波ダッシュで圧縮、不連続は `US-1, US-3, US-5` のようにカンマ区切り。範囲とカンマの混在可(`US-1〜12, US-114`)。横断 US を吸収する場合もここに含める
    - **関連要求 ID**: 当該 UC の主要な要求 ID(`DOM-<コード>-BR/IRR/DATA-*` および/または `DOM-COMMON-*`)。関連 US から逆引きで決定。1 UC は両方の要求体系を参照しうる(横断要求の取り込みも併記される)
6. 並び順: ユーザーストーリー一覧のドメイン並び(意向把握 → 設計書作成 → … → 統制・証跡管理)を踏襲し、最後に横断 UC(認証・中断再開 等)を並べる
7. 上流から確定できない値には要確認マーカーを付ける(例: 主アクター判定の境界・トリガの業務名・[フェーズ2] 境界 等)
8. 将来段階導入の UC には `[フェーズ2]` ラベルを付ける(ユーザーストーリー一覧・PRD・ドメイン要求仕様書の同種ラベルと表記を揃える)
9. ID は **`UC-<連番>` 形式の通し番号**。ドメインや区分は ID に含めず「ドメイン / 区分」列に降ろす(CLAUDE.md §8 の「単一種類しか振らない成果物では `<成果物略号>-<連番>`」に準拠)

## 留意事項

- **本書はフラット表のみ**: 「本書について」「想定読者」「注記」「受け入れ基準」 等の編集メタ章を持たない。タイトル直下に単一の表のみを置く。前提・編集チェック観点は本スキル(および テンプレ冒頭 HTML コメント)に集約する
- **ドメイン別の章分割をしない**: スプレッドシート/Excel 移送を想定したフラット構造を厳守する。`### {{ドメイン名}}` のような subsection を復活させない。区別は表の「ドメイン / 区分」列で表現する
- **US 多 : UC 1**: 1 UC に複数の US が紐づくのが正(その逆はない)。1 US が複数 UC に登場する横断 US(通知・帳票・同意・縮退・例外UX 等)は、最も関連の強い UC に主として配置し、必要に応じて他 UC からも参照される
- **業務局面 × 主アクター が分割軸**: 「ドメイン」だけでも「アクター」だけでも分割しない。業務局面(意向把握 / 申込受付 / 引受査定 …)と 主アクター(募集人 / 事務 / コンプライアンス部 …)の組合せが UC の単位。同一業務局面でも主アクターが業務責務的に異なるなら別 UC とする
- **シーケンス図分割単位としての粒度**: D2/D3 でシーケンス図が描けるよう、各 UC は主アクター・トリガ・成果が一意に決まる粒度であること。1 UC に「申込受付」と「停留申込収束」のように業務責務が異なるシナリオを兼任させない
- **横断 UC は限定**: 横断 UC は「それ自体が独立した業務局面となるもの」のみ(認証 / 中断・再開 等)。通知・帳票・同意取得/撤回・外部連携障害時縮退・例外UX は業務局面 UC のサブシナリオとして「関連 US」列に取り込み、独立 UC 化しない(横断 UC を野放図に増やすと業務局面 UC のシーケンス図から横断 UC への参照ノイズが過剰になる)
- **業務言語に留まる**: UC 名・トリガ・主要価値のいずれも業務言語で書く。機能名・画面名・テーブル名・API 名等の How(D2 以降)は混入させない。「業務」の観点に立ち、システム実装が想起される語に「業務」を冠する(例: 状態→業務状態、例外→業務運用イレギュラー対応)
- **アクター ID は二重採番しない**: アクター一覧の `ACT-*` をそのまま使う(本書独自の `USR-*` 等を作らない)。主アクター列の表記は `アクター名: ACT-N` のコロン区切り(複数は中黒)
- **US ID 範囲表記の規律**: 関連 US 列で連続範囲は `US-1〜12` のように波ダッシュで圧縮、不連続は `US-1, US-3` のようにカンマ区切り。範囲とカンマの混在(`US-1〜12, US-114`)を許容する。範囲表記時も含まれる全 US が当該 UC に紐づくことを必ず確認する(範囲内に他 UC 所属 US が混在しないこと)
- **網羅性はスキル側で担保**: 「全 US が少なくとも 1 つの UC に割り当たっているか」「全直接利用アクターに 1 つ以上の UC が主アクターとして割り当たっているか」等の網羅性チェックは本スキル「完成時のセルフチェック」で担保し、成果物本体に編集チェック章を設けない
- **フェーズ2 ラベル**: 初期フェーズ対象外の UC には `[フェーズ2]` ラベルを付ける。ユーザーストーリー一覧・PRD・ドメイン要求仕様書の同種ラベルと表記を揃える
- **上流との整合性**: ユーザーストーリー一覧の `US-*` ID と関連 US 列の範囲表記が完全一致すること(参照切れ・取りこぼしを起こさない)。各 UC の関連要求 ID が上流要求書に実在することを確認する

## 完成時のセルフチェック

執筆完了時に以下の観点で出力を検査し、不備があれば修正したうえで出力を完了とする。

- 「本書について」「想定読者」「注記」「受け入れ基準」 等の編集メタ章を復活させていないか(タイトル直下が単一表のみになっているか)
- ドメイン別の `### {{ドメイン名}}` subsection を作っていないか(フラット表 1 枚に集約されているか)
- ユーザーストーリー一覧の全 US ID(US-1 〜 US-最終)が、少なくとも 1 つの UC の「関連 US」列に含まれているか(取りこぼしがないか)
- 全直接利用アクター(PRD §「体験設計 > システム利用ユーザー」配下)が、少なくとも 1 つの UC で主アクターまたは関連アクターとして登場しているか
- UC 件数が 15-25 件の目安に収まっているか(超過時は集約不足、不足時は業務局面の取りこぼしの可能性を点検)
- 各 UC が「主アクター 1(または同等業務責務の複数)・トリガ 1・主要価値 1」を満たし、シーケンス図が独立して描ける粒度になっているか
- 1 UC に業務責務が異なる複数業務シナリオを兼任させていないか(兼任があれば UC を分割しているか)
- 横断 UC が「それ自体が独立した業務局面となるもの」のみ(認証・中断再開 等)に限定され、通知・帳票・同意・縮退・例外UX が独立 UC 化されていないか(=業務局面 UC の関連 US 列に取り込まれているか)
- UC 名が「{{目的語}}を{{動詞}}する」の能動形になっているか、システム実装語(画面名・API 名・テーブル名)が混入していないか
- 主アクター列が `アクター名: ACT-N` 形式で、アクター一覧の ID と一致しているか(本書独自のユーザー ID が作られていないか)
- 関連 US の範囲表記(`US-1〜12` 等)が連続する US 群でのみ用いられ、範囲内に他 UC 所属 US が混在していないか
- 関連要求 ID が、関連 US の「紐づく要求 ID」列の和集合の主要部分と整合しているか(参照切れ・齟齬がないか)
- 並び順がユーザーストーリー一覧のドメイン並びを踏襲し、最後に横断 UC が連続して並んでいるか
- ID が `UC-<連番>` の通し番号で重複なく連続しているか(ドメイン情報を ID に含めていないか)
- 将来要件に `[フェーズ2]` ラベルが付与され、上流の同種ラベルと表記が揃っているか
- `{{` や XXX のプレースホルダ、不要な要確認マーカーが残っていないか

## 下流への伝搬関係

作成・更新時は以下の下流成果物との整合を意識する:

| 下流成果物 | ユースケース一覧から何を引く |
|---|---|
| 機能一覧(D2) | 各 UC から「主要価値の実現に必要な機能」を抽出。横断 UC(認証・中断再開)由来の機能は横断機能としてまとめる |
| 画面要件群(D2) | 各 UC の主アクター × 主要価値 から、主アクター視点での画面要件を導出 |
| L1/L2 シーケンス図(D2/D3) | **本書の 1 UC = 1 シーケンス図** を基本とする。主アクター・トリガ・主要価値が一意のためシーケンスがそのまま描ける |
| テスト計画 | 各 UC の主要価値・トリガ・関連 US の Acceptance Criteria を起点に業務観点テストケースを設計 |

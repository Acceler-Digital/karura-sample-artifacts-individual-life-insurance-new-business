# プロジェクト共通の慣習

本リポジトリで Claude を介して成果物を生成・更新する際の共通ルール。
個別の成果物固有のルール(章構成・記述粒度・上下流関係など)は `.claude/skills/<skill-name>/SKILL.md` を参照する。

## 1. プレースホルダ書式

未確定の値・プロジェクト固有の値で置換すべき箇所は二重波括弧で囲む:

```markdown
{{xxx}}
```

成果物完成時には実プロジェクトの値で置換し、`{{` が残らない状態にする。

## 2. HINT記法(テンプレート内の書き方ガイド)

テンプレート内の記入指針は HTML コメントとして埋め込む。記入完了後は削除して構わない:

```markdown
<!-- HINT: ここに何を書くかの指針 -->
```

### 注意: HTML コメント内に `-->` を含めない

HTML コメントは最初に出現する `-->` でクローズされる仕様のため、HINT 本文中に矢印記法(`-->`、`-.->` 等)を書くと、その時点でコメントが終了し、以降の文字列が本文として描画されてしまう。

Mermaid・PlantUML・シーケンス図・状態遷移図など、矢印を伴う記法を HINT 内で説明したい場合は以下のいずれかで対処する。

- 矢印記号を使わず **説明的な表現** に置き換える(例:「実線矢印」「破線矢印」)
- 具体的な記法はテンプレート本体側のコードブロックに記述し、HINT 内では参照に留める

```markdown
<!-- ❌ HINT 内に `-->` を直接書くと、HTMLコメントがそこで閉じる -->
<!-- HINT:
凡例:
  - `-->`: 業務工程上の遷移
  - `-.->`: 横断ドメインへの参照
-->

<!-- ✅ 矢印記号を説明的に置き換え、具体記法はコードブロックに委ねる -->
<!-- HINT:
凡例(具体的な記法は下記コードブロック参照):
  - 実線矢印: 業務工程上の遷移
  - 破線矢印: 横断ドメインへの参照
-->
```

### HINT と本文の役割分担

HINT と本文(テンプレート本体に書かれる文章・表・箇条書き 等)は役割を分けて書く。両者の内容が重なるのは設計上の不備とみなす。

| 本文の性質 | HINT の役割 |
|---|---|
| プレースホルダ主体(大半の章) | リッチに記述(何を書くか・典型例・境界線・代表例カテゴリ 等) |
| 完全な定型文(§本書について 等) | ミニマル(置換すべきプレースホルダの示唆のみ)、または削除 |

**ルール**: HINT は本文を埋める前に読むガイドであり、本文の言い直しではない。本文に書かれた内容を HINT で繰り返さない。表のセルに具体例が書かれているなら HINT で同じ具体例を列挙しない。

## 3. 要確認マーカー(推測・仮置き箇所の明示)

Claude が推測・仮置きで埋めた箇所には可視マーカーを残す。
**ブロック記法**と**インライン記法**を用途で使い分ける。

### ブロック記法

段落全体・セクション全体に対する仮置き、または複数行の詳細説明が必要な場合に使用:

```markdown
> ⚠️ **要確認**: {{確認内容 / 仮置きしている前提}}
```

### インライン記法

テーブルセル内、文中の単一値、固有名詞ピンポイントなど、特定箇所だけを示したい場合に使用:

```markdown
{{値}}【要確認】
{{値}}【要確認: 短い理由】
```

例:
```markdown
| 申込から契約成立までのリードタイム | 5営業日以内(現行比50%短縮)【要確認】 | 業務データ分析 | 月次 |
```

### 使い分けの指針

| ケース | 記法 |
|---|---|
| 段落全体・セクション全体に対する仮置き / 複数行の説明 | ブロック記法 |
| テーブルセル内 | インライン記法 |
| 文中の特定値・固有名詞ピンポイント | インライン記法 |

両形式とも `grep '要確認'` で一括検出できる。

### 適用対象

- 全フェーズ・全成果物(D0〜D9 すべて)
- スキル経由・直接対話を問わず、Claude が生成・更新するすべての文書

### Claude の振る舞い

1. 提供された情報で書ける箇所は埋める
2. 推測・仮置きで埋めた箇所には必ず要確認マーカーを付ける
3. 重要数値(KPI 目標値、期日、予算上限、法令適用範囲、リスク評価 等)を推測で書く際は必ずマーカーを付ける
4. マーカーが残っている間は未完成扱い

### マーカーの解消

- 人手で内容を確認・修正後、ユーザーがマーカー部分を削除する
  - ブロック記法: ブロッククォート行ごと削除
  - インライン記法: `【要確認】` または `【要確認: ...】` の部分のみ削除
- マーカー残存は CI で fail させない方針(手元・PR コメント等で気づける程度の警告に留める)
- 残存状況の確認は `pnpm check:markers` で実行

## 4. ファイル配置の規約

| 種類 | 置き場所 |
|---|---|
| 成果物本体 | `docs/<phase>/<artifact-name>.md` |
| 成果物スキル | `.claude/skills/<phase>-<artifact-name>/SKILL.md` |
| 成果物テンプレート | `.claude/skills/<phase>-<artifact-name>/template.md` |

`<phase>` は `D0.project-management`, `D1.business-requirements`, `D2.system-requirements` などのフェーズ別ディレクトリ名。
スキル名は `<phase>-<artifact-kebab>` 形式(例: `d1-business-requirement-document`)。
テンプレートはスキルと同一ディレクトリに同梱し、SKILL.md から相対参照する。本リポジトリは Claude を中核に据える前提のため、テンプレートはスキルが読み込む内部リソースとして扱う。

### template / SKILL / CLAUDE の責務分担

三者は内容を重複させず、書く場所を明確に分ける。同じ内容が二箇所に現れるのは設計上の不備とみなす。

| 置き場所 | 書くもの | 書かないもの |
|---|---|---|
| **template.md** | 文書の骨組みのみ — タイトル・章/表の構成(列名・並び)・プレースホルダ行。冒頭 HTML コメントは SKILL.md / CLAUDE.md への参照案内に留める | 振る舞いルール(埋め方・粒度・分割/集約の判断・チェック観点)、汎用規約の再掲 |
| **SKILL.md** | 当該成果物**固有**の振る舞い — 生成トリガ、input/output、記述粒度、分割/集約単位、各列の埋め方 等 | description と重複する役割/位置づけ、template に従えば自明な章構成、他成果物共通の汎用規約(下記 CLAUDE.md に集約) |
| **CLAUDE.md** | 全成果物**横断**の汎用規約 — プレースホルダ書式・HINT記法・要確認マーカー・文体・ID 表記・サイドバー登録・コミット規約 等 | 個別成果物固有のルール |

**原則**: テンプレートはあくまで骨組み。振る舞いの制御は SKILL.md、汎用的な話は CLAUDE.md に書く。判断に迷ったら「この記述は他の成果物でも成り立つか?(=CLAUDE.md)」「この成果物だけの振る舞いか?(=SKILL.md)」「埋めるべき器の形か?(=template.md)」で振り分ける。

### SKILL.md の書き方(重複を持たない)

SKILL.md には「その成果物を生成するために実際に必要な固有の振る舞い」だけを書く。各情報は一意の置き場所を持ち、本文に一度だけ現れる。以下は **書かない**:

- **description と重複する内容**: 役割・位置づけ・発火条件は description が持つ。本文冒頭に成果物の紹介文・位置づけ段落を置かず、description にない振る舞い(生成トリガ・粒度・各列の埋め方・固有の判断基準)だけを書く。
- **章構成セクション**: 文書の構造(章立て・表の列名と並び)は template.md の骨組みに従うため自明。`## 章構成` を設けず、SKILL は「どう埋めるか」だけを書く。
- **自明な input/output**: 更新時に当該成果物自身を読む、等の自明な項目は列挙しない。
- **同一ルールの再掲**: 1 つの規律は最も適切な 1 箇所にだけ書く。同じ内容を「留意事項」「完成時のセルフチェック」「下流への伝搬関係」等で言い直さない。チェック観点はルール本文に内在させ、別立てのチェックリストにしない。
- **定型セクションの惰性設置**: 「本書について」「留意事項」「セルフチェック」「下流伝搬」等を型として埋めない。対応する固有の振る舞いがなければ節を作らない。

## 5. Docusaurus サイドバーへの登録

`docs/` 配下に新しいファイルやフォルダを追加・リネーム・削除した場合は、リポジトリ直下の `sidebars.ts` を必ず合わせて更新する(登録しないとドキュメントサイト上から参照できない)。

- **新フェーズ用フォルダ追加時**(例: `D1.business-requirements/` を新設): 新しい `category` ブロックを追加する。`label` はフェーズ名(例: `D1. ビジネス要求`)
- **既存フェーズに新ファイル追加時**: 対応する `category` の `items` に `{ type: "doc", id: "<phase>/<file-stem>", label: "<日本語タイトル>" }` を追記する
- `id` は `docs/` からの相対パスから拡張子を除いた値(例: `D1.business-requirements/business-requirement-document`)
- `label` は人間が読む日本語タイトル(例: `ビジネス要件定義書`)
- 並び順は AIDD フロー順(D0 → D1 → D2 → ...)を維持し、フェーズ内では成果物フロー上の依存順を尊重する

スキル経由・直接対話を問わず、Claude が `docs/` を編集するすべての作業に適用される。

## 6. 参照元の境界

Claude が成果物を生成・更新する際、**明示的な指示がない限り** 以下のものを参照源として扱ってはならない。

### 既定で参照しないもの

- **D0 配下のすべてのファイル**(プロジェクトインデックス、成果物フロー、体制図、課題・リスク管理表 等のプロジェクト管理メタデータ)

### 理由

- D0 は本来 D1 以降の成果物から派生する情報を運営用に集約したもの。逆方向で参照すると循環参照となり、ステークホルダー未合意の暫定値が「既知の事実」として下流に伝播するリスクがある
- D0 にはプロジェクト立ち上げ初期の仮置き値が残っていることがあり、合意済みとは限らない

### 既定で参照してよいもの

- ユーザーとの対話で明示的に提示された情報
- 対象成果物の正規の上流成果物(例: D2 を作る際の D1)
- ユーザーが「参照してよい」と明示指示したファイル

### 振る舞い

- D0 等にしか書かれていない情報を使いたくなった場合は、ユーザーに参照可否を確認する
- 該当情報がない場合は、空欄にせず推測値+要確認マーカーで記述する

## 7. 文体・言い回しの規約

成果物の本文(章導入文・前文・補足説明など読み物として続く段落)は **です・ます調** で統一する。
表のセル内・箇条書きの定義/項目記述・短いラベル類は **常体(する調)** を用いてよい。

### 適用対象

- 全フェーズ・全成果物(D0〜D9 すべて)
- スキル経由・直接対話を問わず、Claude が生成・更新するすべての文書

### 使い分けの指針

| 箇所 | 文体 |
|---|---|
| 「本書について」等の前文・概要段落 | です・ます調 |
| 章の導入文・補足の説明文(リスト/表の前後にある段落) | です・ます調 |
| 表のセル内 | 常体 |
| 箇条書きの定義・項目記述 | 常体 |
| 見出し・短いラベル | 不問 |

### 例

```markdown
<!-- ✅ 説明段落: です・ます調 -->
本書は、本システムに関わる全アクターを一覧化したドキュメントです。

<!-- ✅ 章の導入文: です・ます調 -->
アクターは以下3軸で分類します。

<!-- ✅ 表セル内: 常体 -->
| ACT-1 | 営業職員 | ... | 設計書作成・申込受付を実施する | ... |

<!-- ❌ 説明段落で常体 -->
本書は、本システムに関わる全アクターを一覧化したドキュメントである。
```

### Claude の振る舞い

1. 上流の正規参照源(例: BRD)が存在する場合は、その説明段落の文体に揃える
2. 上流に該当文体例がない場合は **です・ます調** をデフォルトとする
3. 表セル内・箇条書き項目内の述部は常体で簡潔に書く。説明段落と一文中で文体が混ざらないよう、段落と一覧は明確に分けて記述する

## 8. ID の表記規約

成果物の表・リスト等で各エントリに付与する ID は、先頭に **成果物略号** を冠する **`<成果物略号>-<カテゴリ>-<連番>` 形式** とする。成果物略号を冠することで「どの成果物が振った ID か」を参照箇所だけで判別でき、下流成果物からの相互参照・トレーサビリティが容易になる。

### 形式

```
<成果物略号>-<カテゴリ>-<連番>
```

- **成果物略号**: その ID を定義した成果物を表す大文字略号(例: `PRD` / `BRD` / `ACT` / `DOM`)。AIDD フェーズ名(`D0`, `D1`, `D2` 等)と衝突する単一文字は使わない
- **カテゴリ**: 1成果物が複数種類の ID を振る場合の種別コード(大文字)。単一種類しか振らない成果物では省略し `<成果物略号>-<連番>` とする。カテゴリに親子関係がある場合はハイフンで連結してよい(例: セキュリティ要求配下のデータアクセス要求 → `DOM-COMMON-SEC-DATA-1`)
- **連番**: 1始まりの整数。ゼロ埋めしない。連番の代わりに意味コード(ニーモニック)を用いる確立済み語彙(ドメインコード 等)はそれを許容する
- 区切りはハイフン `-`。英字部はすべて大文字

### 成果物略号

| 成果物 | 略号 | カテゴリ / 連番の例 |
|---|---|---|
| ビジネス要件定義書(BRD) | `BRD` | `BRD-KPI-1` / `BRD-RISK-1` 等 |
| プロダクト要求仕様書(PRD) | `PRD` | 方針・概念・制約・受け入れ基準を扱う。**原則として要求 ID は付さない**(具体的な要求はドメイン共通要求仕様書・ドメイン要求仕様書群に降ろす) |
| アクター一覧 | `ACT` | 単一種類のためカテゴリ省略。`ACT-1` |
| ドメイン定義書 | `DOM` | ニーモニックを連番の代わりに用いる。`DOM-HEAR` / `DOM-APPL` 等。各エントリはドメイン共通要求仕様書・ドメイン要求仕様書群の親キーとなる |
| ドメイン共通要求仕様書 | `DOM-COMMON` | 全ドメインに横断的に効く要求。`DOM-COMMON-FR-1`(機能要求)/ `DOM-COMMON-NFR-1`(非機能)/ `DOM-COMMON-UX-1`(UI/UX 方針)/ `DOM-COMMON-SEC-1`(セキュリティ)/ `DOM-COMMON-SEC-DATA-1`(データアクセス: セキュリティ要求配下の入れ子カテゴリ)/ `DOM-COMMON-REG-1`(法務・コンプラ) |
| ドメイン要求仕様書群(個別) | `DOM-<ドメインコード>` | ドメイン定義書のニーモニック(`HEAR` / `APPL` 等)を冠する。`DOM-HEAR-BR-1`(業務ルール)/ `DOM-HEAR-IRR-1`(業務運用イレギュラー)/ `DOM-HEAR-DATA-1`(データアクセス) 等 |
| ユーザーストーリー一覧 | `US` | 単一種類・通し連番。`US-1`, `US-2`, ... |
| 画面コンセプト集 | `SC` | `SC-PR-1`(体験原則)/ `SC-SCENE-1`(代表シーン) |
| 機能一覧 | `FUNC` | 単一種類・通し連番。`FUNC-1`, `FUNC-2`, ... |
| 非機能要件 | `NONFUNC` | 単一種類・通し連番。`NONFUNC-1`, `NONFUNC-2`, ...(`FUNC` と対で読めるよう、冗長でも略さない) |
| 外部システム一覧 | `EXT` | 単一種類・通し連番。`EXT-1`, `EXT-2`, ...(本システムが連携する外部システム/外部サービス)。C4 モデル等は本一覧の `EXT-*` を流用する |

**ID 先頭でカテゴリ判別:** ID の先頭部分を見るだけで「どのカテゴリの ID か」が分かる対称性を維持する。`PRD-*`(方針)/ `DOM-COMMON-*`(ドメイン共通要求)/ `DOM-<コード>-*`(ドメイン個別要求)/ `BRD-*`(事業要求)/ `ACT-*`(アクター)/ `US-*`(ユーザーストーリー)/ `SC-*`(画面コンセプト)/ `FUNC-*`(機能)/ `NONFUNC-*`(非機能)/ `EXT-*`(外部システム) のように、ID 体系を統一する。

他成果物の略号・カテゴリは当該成果物のテンプレート/スキルで定義する。

### ID の列挙・範囲表記

成果物の「関連 ID」列など、複数 ID を 1 セルに列挙する場合の表記を統一する。

- **連続範囲は波ダッシュ `〜` で圧縮**: `UC-1〜3`(= UC-1, UC-2, UC-3)
- **不連続・複数はカンマ区切り**: `US-1, US-3, US-5`
- **範囲とカンマの混在可**: `US-1〜12, US-114`
- 範囲表記時は、範囲内のすべての ID が当該行に紐づくことを確認する(範囲内に無関係な ID を含めない)

### 他成果物が定義した ID の参照

- 他成果物が定義した ID(`ACT-*` / `UC-*` / `DOM-*` 等)は **そのまま引用** し、参照側で独自に振り直さない(二重採番の禁止)
- 人物・アクターを参照する際は **`名称: ID`** のコロン区切り、複数は中黒で併記(例: `営業職員: ACT-1`、`募集人: ACT-1・ACT-2`)

### 適用対象

- 全フェーズ・全成果物(D0〜D9 すべて)
- スキル経由・直接対話を問わず、Claude が生成・更新するすべての文書

### 既存成果物の移行

- 旧形式(成果物略号なし・ハイフンなし。例: `ACT1`)で記述済みの成果物は、**当該成果物を次に更新する際に新形式へ寄せる**(一括改名は必須としない)
- 移行時は、その ID を参照している下流成果物の表記も同時に更新し、参照不整合を残さない

### 例

```markdown
<!-- ✅ 成果物略号 + (カテゴリ) + 連番、英字大文字、ハイフン区切り -->
| DOM-COMMON-FR-1 | 業務通知 | ... |
| DOM-COMMON-NFR-3 | 可用性 | ... |
| DOM-HEAR-BR-1 | 意向把握項目の網羅 | ... |
| DOM-HEAR-IRR-2 | 聴取の長時間中断 | ... |
| ACT-1 | 営業職員 | ... |
| DOM-HEAR | 意向把握 | ... |
| US-1 | ... | ... |

<!-- ❌ 成果物略号なし / 小文字 / ハイフンなし -->
| FR1 | 認証 | ... |
| dom-common-fr-1 | 認証 | ... |
| HEARBR1 | 認証 | ... |
```

### 検出

`grep -oE '<成果物略号>-'`(例: `grep -oE 'DOM-COMMON-'` / `grep -oE 'DOM-HEAR-'`)で成果物別に ID を一括抽出できる。レイヤー単位の集計は `grep -oE '^DOM-'`(ドメインレイヤー全般)で取得できる。

## 9. Git コミット規約

Claude がコミットを作成する際は以下を厳守する。

### コミットメッセージに署名・帰属トレーラーを入れない

コミットメッセージ(本文・フッタを含む)に、AI による生成・補助を示すトレーラーや署名を **一切付与しない**。具体的には以下を含めない。

- `Co-Authored-By: Claude ...`(および Claude/AI を示す任意の `Co-Authored-By`)
- `Generated with ...` / `🤖 Generated with ...` 等の生成ツール表記
- その他、Anthropic・Claude・AI アシスタントを示す帰属文・絵文字・宣伝文言

コミットメッセージは **変更内容そのものの説明のみ** で構成する。本リポジトリの既存履歴に倣い、日本語の Conventional Commits 形式(`feat:` / `refactor:` / `doc:` 等)を用いる。

### コミット対象は当該作業で変更したファイルに限定する

`git commit` 時は **今回の作業で実際に変更したファイルのみ** を明示的なパス指定でコミットする。作業ツリーに無関係な変更やステージ済みファイルがあっても巻き込まない。`git commit --amend` をパス指定なしで実行すると既存ステージ内容を巻き込むため、対象を限定したい場合は必ずパスを明示する。

## 10. スキル description の記述方針

成果物スキル(`.claude/skills/<skill-name>/SKILL.md`)の frontmatter `description` は **最小限** に保つ。`description` の唯一の役割は **ルーティング判定**(モデルが全スキルの description だけを読み比べ、どれを発火すべきか決める)であり、判定に効かない情報を盛るとノイズになって発火精度を下げる。

### description に書くもの(これだけ)

- **役割**: 何をする成果物か(例: 「機能一覧の新規作成・更新・レビューを支援する」)
- **発火条件**: いつ呼ぶか(例: 「ユースケース一覧の策定後、機能の洗い出し段階で呼び出す」)
- **区別キー**: 似た成果物スキルと取り違えない最小限のキーワード(例: フェーズ記号 `D2`・成果物名・直近の上流成果物名)

目安は **1〜3 文**。

### description に書かないもの(本文へ)

以下は SKILL.md 本文が読み込まれて初めて効くため、description には載せない。

- 上流下流・上位下位関係の説明、章構成、列構成
- 記述粒度・分割単位・ID 体系・スコープ境界
- 下流成果物への伝搬関係、完成時チェック観点

これらは SKILL.md 本文に正規の置き場所がある(§4「SKILL.md の書き方」を参照)。description はそこへ誘導する看板であり、内容の要約ではない。

### 適用対象・移行

- 全フェーズ・全成果物スキル(D0〜D9)に適用する
- 既に長い description を持つ既存スキルは、**当該スキルを次に更新する際に最小形へ寄せる**(一括改修は必須としない)

## 11. 成果物生成の共通方針

全フェーズ・全成果物の生成・更新に共通して適用する。個別成果物の SKILL.md ではこれらを再掲せず、当該成果物固有の適用(どの上流のどの章を引くか・どの列に何を書くか 等)だけを書く。

### 上流からの直接生成

- 正規の上流成果物が存在する成果物は、新規作成時に **ユーザーへのヒアリングを行わず**、上流から直接生成する。推測・仮置きした箇所には要確認マーカー(§3)を付す
- **例外は最上流の成果物のみ**(上流が存在しない BRD 等)。この場合に限りヒアリングで起こす

### フェーズの抽象度(下流の How を先取りしない)

各成果物は自身のフェーズの抽象度に留まり、下流フェーズで決めるべき事項(How)を先取りしない。実装を想起させる語が出てきたら、それは下流に降ろすべき兆候。

| フェーズ | 留まるべき抽象度 | 書かない(下流の How) |
|---|---|---|
| D1(ビジネス要求) | 業務言語。システム実装を想起させる語には「業務」を冠して中立化(例: 状態遷移→業務状態遷移、例外処理→業務運用イレギュラー対応) | 機能名・画面・API・データ等のシステム/設計用語 |
| D2(システム要件) | システムが提供する機能能力(capability) | 画面名・API 名・テーブル名・カラム名・実装技術・アルゴリズム 等の D3 実装語 |
| D3(システム設計)以降 | 設計・実装の How | — |

### 上下流の一貫性

- **トレーサビリティ(全件網羅)**: 上流が定義した ID は、下流成果物の「関連 ID」列で取りこぼさず参照する。上流の全 ID が下流のいずれかの行に少なくとも 1 回現れることを確認する
- **段階導入ラベルの整合**: `[フェーズ2]` 等の段階導入ラベルは、上流の同種ラベルと表記を揃え、上流で当該ラベルが付く項目由来のものに付与する

# プロジェクト共通の慣習

本リポジトリで Claude を介して成果物を生成・更新する際の共通ルール。
個別の成果物固有のルール(章構成・記述粒度・上下流関係など)は `.claude/skills/<skill-name>/SKILL.md` を参照する。

## 1. プレースホルダ書式

未確定の値・プロジェクト固有の値で置換すべき箇所は二重波括弧で囲む:

```markdown
{{xxx}}
```

成果物完成時には実プロジェクトの値で置換し、`{{` が残らない状態にする。

## 2. HINT記法(テンプレート内の書き方ガイド)

テンプレート内の記入指針は HTML コメントとして埋め込む。記入完了後は削除して構わない:

```markdown
<!-- HINT: ここに何を書くかの指針 -->
```

## 3. 要確認マーカー(推測・仮置き箇所の明示)

Claude が推測・仮置きで埋めた箇所には可視マーカーを残す。
**ブロック記法**と**インライン記法**を用途で使い分ける。

### ブロック記法

段落全体・セクション全体に対する仮置き、または複数行の詳細説明が必要な場合に使用:

```markdown
> ⚠️ **要確認**: {{確認内容 / 仮置きしている前提}}
```

### インライン記法

テーブルセル内、文中の単一値、固有名詞ピンポイントなど、特定箇所だけを示したい場合に使用:

```markdown
{{値}}【要確認】
{{値}}【要確認: 短い理由】
```

例:
```markdown
| 申込から契約成立までのリードタイム | 5営業日以内(現行比50%短縮)【要確認】 | 業務データ分析 | 月次 |
```

### 使い分けの指針

| ケース | 記法 |
|---|---|
| 段落全体・セクション全体に対する仮置き / 複数行の説明 | ブロック記法 |
| テーブルセル内 | インライン記法 |
| 文中の特定値・固有名詞ピンポイント | インライン記法 |

両形式とも `grep '要確認'` で一括検出できる。

### 適用対象

- 全フェーズ・全成果物(D0〜D9 すべて)
- スキル経由・直接対話を問わず、Claude が生成・更新するすべての文書

### Claude の振る舞い

1. 提供された情報で書ける箇所は埋める
2. 推測・仮置きで埋めた箇所には必ず要確認マーカーを付ける
3. 重要数値(KPI 目標値、期日、予算上限、法令適用範囲、リスク評価 等)を推測で書く際は必ずマーカーを付ける
4. マーカーが残っている間は未完成扱い

### マーカーの解消

- 人手で内容を確認・修正後、ユーザーがマーカー部分を削除する
  - ブロック記法: ブロッククォート行ごと削除
  - インライン記法: `【要確認】` または `【要確認: ...】` の部分のみ削除
- マーカー残存は CI で fail させない方針(手元・PR コメント等で気づける程度の警告に留める)
- 残存状況の確認は `pnpm check:markers` で実行

## 4. ファイル配置の規約

| 種類 | 置き場所 |
|---|---|
| 成果物本体 | `docs/<phase>/<artifact-name>.md` |
| 成果物テンプレート | `templates/<phase>/<artifact-name>-template.md` |
| 成果物スキル | `.claude/skills/<phase>-<artifact-name>/SKILL.md` |

`<phase>` は `D0.project-management`, `D1.business-requirements`, `D2.system-requirements` などのフェーズ別ディレクトリ名。
スキル名は `<phase>-<artifact-kebab>` 形式(例: `d1-business-requirement-document`)。

## 5. Docusaurus サイドバーへの登録

`docs/` 配下に新しいファイルやフォルダを追加・リネーム・削除した場合は、リポジトリ直下の `sidebars.ts` を必ず合わせて更新する(登録しないとドキュメントサイト上から参照できない)。

- **新フェーズ用フォルダ追加時**(例: `D1.business-requirements/` を新設): 新しい `category` ブロックを追加する。`label` はフェーズ名(例: `D1. ビジネス要求`)
- **既存フェーズに新ファイル追加時**: 対応する `category` の `items` に `{ type: "doc", id: "<phase>/<file-stem>", label: "<日本語タイトル>" }` を追記する
- `id` は `docs/` からの相対パスから拡張子を除いた値(例: `D1.business-requirements/business-requirement-document`)
- `label` は人間が読む日本語タイトル(例: `ビジネス要件定義書`)
- 並び順は AIDD フロー順(D0 → D1 → D2 → ...)を維持し、フェーズ内では成果物フロー上の依存順を尊重する

スキル経由・直接対話を問わず、Claude が `docs/` を編集するすべての作業に適用される。

## 6. 参照元の境界

Claude が成果物を生成・更新する際、**明示的な指示がない限り** 以下のものを参照源として扱ってはならない。

### 既定で参照しないもの

- **D0 配下のすべてのファイル**(プロジェクトインデックス、成果物フロー、体制図、課題・リスク管理表 等のプロジェクト管理メタデータ)

### 理由

- D0 は本来 D1 以降の成果物から派生する情報を運営用に集約したもの。逆方向で参照すると循環参照となり、ステークホルダー未合意の暫定値が「既知の事実」として下流に伝播するリスクがある
- D0 にはプロジェクト立ち上げ初期の仮置き値が残っていることがあり、合意済みとは限らない

### 既定で参照してよいもの

- ユーザーとの対話で明示的に提示された情報
- 対象成果物の正規の上流成果物(例: D2 を作る際の D1)
- ユーザーが「参照してよい」と明示指示したファイル

### 振る舞い

- D0 等にしか書かれていない情報を使いたくなった場合は、ユーザーに参照可否を確認する
- 該当情報がない場合は、空欄にせず推測値+要確認マーカーで記述する
